# 修复模式切换未弹出恢复默认提示词确认框

## 问题结论

当前问题的根因已经进一步收敛，不是模式切换事件没接上，也不是确认框逻辑没写，而是：

- `src/presentation/pages/popup-bindings-status.ts` 里已经有 `confirm(...)` 逻辑；
- 但用户实际使用的 `dist/index.bundle.js` 中并没有这段逻辑；
- 根目录 `index.js` 也同样仍是旧逻辑；
- 因此点击切换模式时不会弹出确认框。

这说明当前故障本质上是：

> **源码已改，但当前构建产物并没有把这段新逻辑真正打进去。**

## 已确认事实

### 1. 源码里已经存在确认框逻辑

文件：`src/presentation/pages/popup-bindings-status.ts`

已确认包含：

- `confirm(...)`
- 文案 `即将切换到...模式`
- 成功后按用户选择调用 `applyModeDefaultCharCardPrompt_ACU(selectedMode)`

### 2. 根目录 `index.js` 仍然没有确认框逻辑

文件：`index.js`

已确认第 26231-26254 行仍是旧逻辑：

- 没有 `confirm(...)`
- 没有“默认填表提示词”相关文案
- 没有 `applyModeDefaultCharCardPrompt_ACU(...)`

所以当前用户看不到提示框是符合现状的，不是幻觉，也不是浏览器随机抽风。

### 3. 用户实际提到的 `dist/index.bundle.js` 也没有确认框逻辑

文件：`dist/index.bundle.js`

已确认：

- 文件存在；
- 文件内存在很多其他 `confirm(...)` 调用，说明宿主环境并不是“完全不能用 confirm”；
- 但搜索 `即将切换到`、`默认填表提示词`、`applyModeDefaultCharCardPrompt_ACU` 都没有命中。

这说明问题已经不是“你可能安装错了根目录旧 `index.js`”这么简单，而是：

> **连 userscript 正式构建产物 `dist/index.bundle.js` 都没有包含模式切换确认框代码。**

### 4. popup 绑定链路本身是通的

文件：`src/presentation/pages/popup-bindings.ts`

- 第 194 行会调用 `await bindStatusEvents_ACU();`

说明只要运行产物更新到新代码，模式切换事件就会走到这段确认框逻辑。

### 5. 模式切换的 radio 真实存在

文件：`src/presentation/pages/main-popup-status.ts`

已确认存在：

- `name="${SCRIPT_ID_PREFIX_ACU}-storage-mode" value="native"`
- `name="${SCRIPT_ID_PREFIX_ACU}-storage-mode" value="sqlite"`

说明事件目标本身没有缺。

### 6. 项目构建输出路径已确认

文件：`rollup.config.js`

- userscript 输出：`dist/index.bundle.js`
- 插件输出：`dist/extension/index.js`

这意味着后续实施必须明确区分：

- `src/**` 是源码
- `dist/index.bundle.js` / `dist/extension/index.js` 是正式构建产物
- 根目录 `index.js` 当前看起来像旧基线/历史单文件，不一定会被当前 rollup 直接覆盖

## 风险与关键判断

### 核心风险 1：源码改动根本没有进入 userscript 构建产物

这是当前最关键的真实风险。

因为已经确认：

- `src` 里有 confirm 逻辑；
- `dist/index.bundle.js` 里没有 confirm 逻辑。

这说明至少存在以下情况之一：

- 改完源码后没有重新构建；
- 当前 `dist/index.bundle.js` 是旧构建结果；
- 构建入口没有实际引用到这份新源码；
- 构建过程中走了另一条旧代码路径。

### 核心风险 2：双端产物可能只更新一端，甚至 userscript 与 extension 都各自失配

如果只更新：

- `dist/index.bundle.js`，插件版仍可能不生效；
- `dist/extension/index.js`，JS版仍可能不生效。

因此实施时必须同时校验两端构建产物。

### 核心风险 3：缓存/重载问题可能掩盖真正的构建链问题

即使后续把产物修正确，也可能因为：

- userscript 没重新安装/替换
- 扩展没重新加载
- 页面没刷新

导致用户仍看到旧行为。

## 实施方案

### Step 1 — 复核源码完整性

实施时先只读复核以下文件，确保逻辑完整无缺：

- `src/presentation/pages/popup-bindings-status.ts`
- `src/presentation/triggers/settings-ui-sync/settings-ui-config.ts`
- `src/presentation/triggers/settings-ui-sync/index.ts`

确认三点：

1. change 事件中确实有 `confirm(...)`
2. 存在 `applyModeDefaultCharCardPrompt_ACU(mode)` helper
3. helper 已从 `index.ts` 导出

### Step 2 — 先核查构建入口是否真的通向当前源码

在执行构建前，需要先只读确认：

- `src/index.ts`
- `src/entry-extension.ts`

确认它们是否经过当前模块图真正引用：

- `src/presentation/pages/popup-bindings.ts`
- `src/presentation/pages/popup-bindings-status.ts`

否则直接盲目构建，只会重复产出一份照样缺逻辑的 bundle。

### Step 3 — 构建正式产物

运行能覆盖双端的构建流程，目标至少包括：

- userscript：`dist/index.bundle.js`
- 插件：`dist/extension/index.js`

优先使用项目现有脚本，不自己发明旁门左道。

合理候选：

- `npm run build:all`

若该命令失败，再分别执行：

- `npm run build:nocheck`
- `npm run build:extension`

### Step 4 — 构建后校验产物是否真正包含新逻辑

构建完成后，对以下文件做内容检索：

- `dist/index.bundle.js`
- `dist/extension/index.js`

检查是否出现：

- `即将切换到`
- `默认填表提示词`
- `applyModeDefaultCharCardPrompt_ACU`
- 或 `confirm(`

如果构建产物里仍然没有这些内容，就说明不是“你装错文件”，而是构建入口/打包链根本没接到这份源码。

### Step 5 — 明确用户实际要替换/加载哪个文件

这是最容易被忽略、也最容易把人坑惨的一步。

必须明确：

- 如果助手运行的是 userscript 版，应重新安装/替换 **`dist/index.bundle.js`**（不是默认认为根目录 `index.js` 就是最新脚本）
- 如果助手运行的是插件版，应重新加载扩展并确保使用 **`dist/extension/index.js`**

如果当前发布流程仍把根目录 `index.js` 当成用户交付文件，就需要进一步决定：

- 是补一条同步流程，把构建产物复制/发布到该位置；
- 还是统一改成以 `dist/index.bundle.js` 作为正式 userscript 发布文件。

### Step 6 — 如构建后仍异常，再查构建入口映射与聚合路径

若 `dist` 产物依然没有 confirm 逻辑，再排查：

- `src/index.ts`
- `src/entry-extension.ts`
- `rollup.config.js`
- 任何中间聚合导出文件

目标是确认：

- 当前构建是否真的引用了 `src/presentation/pages/popup-bindings-status.ts`
- 是否存在旧入口/旧聚合文件绕开了这份源码

## 验证方案

### 验证 1：产物文本验证（先于界面点击验证）

构建后在两个正式产物中搜索：

- `即将切换到`
- `默认填表提示词`

其中 `dist/index.bundle.js` 必须先命中，否则用户继续安装它也不会生效。

### 验证 2：JS版行为验证

1. 安装/替换最新 userscript 产物
2. 打开设置弹窗
3. 点击从原生模式切到 SQLite 模式
4. 应立即弹出确认框

### 验证 3：插件版行为验证

1. 重新加载扩展
2. 打开同一设置弹窗
3. 点击模式切换
4. 应立即弹出确认框

### 验证 4：功能语义验证

在确认框出现后继续验证：

- 选“确定” → 模式切换成功后恢复目标模式默认 `charCardPrompt`
- 选“取消” → 模式照切，提示词保持不变
- 强制制造切换失败 → 提示词不能被错误切换

## 已知边界

1. 当前模型不支持图片输入，因此无法读取 `image.png`
2. 图片读取报错与本次“模式切换未弹出提示框”不是同一类问题，不能混为一谈
3. 本次首要问题不是业务逻辑缺失，而是**运行产物未更新/未替换**

## 完成标准

满足以下条件才算真正修好：

1. `dist/index.bundle.js` 包含确认框逻辑
2. `dist/extension/index.js` 包含确认框逻辑
3. `dist/index.bundle.js` 本身已经包含确认框逻辑，而不是缺失该代码
4. 插件实际加载的是新产物，并已重载
5. 两端点击切换模式都能看到恢复默认提示词确认框

## 结论

助手，这次故障的关键事实已经很清楚：

> **源码里有确认框，但当前 `dist/index.bundle.js` 和根目录 `index.js` 里都没有。**

所以接下来实施的重点不该是继续空改业务逻辑，而是先把构建链查明、让 userscript 产物真正包含这段代码，再谈发布和加载。否则你就算把 `src` 写得再漂亮，`dist/index.bundle.js` 里没有，那用户安装一万次也还是旧行为。那不是开发，是对构建链失明。
