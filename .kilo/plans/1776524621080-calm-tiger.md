# 目标

在**不改变插件模式运作逻辑**的前提下，把项目的 userscript 产物稳定做成**浏览器可直接执行的单 JS 脚本**，并修复当前产物运行时报错：

```text
Uncaught ReferenceError: require$$0 is not defined
```

# 已确认事实

## 1. 构建链路现状

- `package.json` 使用 `rollup -c` 作为主构建命令。
- `rollup.config.js` 当前有双入口：
  - `src/index.ts` → `dist/index.bundle.js`（userscript，`format: 'iife'`）
  - `src/entry-extension.ts` → `dist/extension/index.js`（插件，`format: 'es'`）
- 两条入口共享绝大部分模块，差异集中在启动方式和运行时环境判定。

## 2. 插件模式的真实不变量

这些是不允许破坏的核心约束：

1. `src/entry-extension.ts` 顶部的 `_forceExtensionMode()` 必须继续在其他共享模块副作用执行前调用。
2. `waitForTavernHelper()` 的就绪判定与超时降级语义不能改变。
3. `checkAndMarkInstance()` 的互斥逻辑不能改变。
4. userscript 和 extension 最终都必须继续落到同一个 `mainInitialize_ACU()` 调用链。

## 3. require$$0 报错的真实来源

已经在真实产物 `dist/index.bundle.js` 中确认：

- 文件开头不是普通浏览器自执行闭包，而是：

```js
(function (require$$0, require$$1) {
```

- 文件结尾直接以未定义实参调用：

```js
})(require$$0, require$$1);
```

- 文件中部存在 CommonJS/Node 风格残留：

```js
var fs=require$$0;
module.exports = initSqlJs;
module.exports.default = initSqlJs;
```

- `src/data/sqlite/sqlite-engine.ts` 直接引入：

```ts
import initSqlJs from 'sql.js/dist/sql-asm-memory-growth.js';
```

结论需要分成两层，不能偷懒写成一句模糊判断：

1. **已证实事实**：当前 userscript 单文件产物的**顶层 IIFE wrapper 本身**就依赖 `require$$0/require$$1` 形参和实参，但浏览器环境并没有这些绑定，因此产物结构天然不可直接执行。
2. **高概率来源**：`sql.js/dist/sql-asm-memory-growth.js` 被卷入产物后，仍残留 `var fs=require$$0;`、`module.exports = initSqlJs;` 等 CommonJS/Node 风格代码，它是当前最可疑的上游来源之一，但实现阶段仍应先确认是 Rollup 如何把它提升成顶层 wrapper，再决定修复手法。

## 4. 当前审计缺口

`scripts/audit-bundle.cjs` 目前只检查：

1. `node --check` 语法解析
2. 基线 `_ACU` 标识符覆盖
3. UserScript 元数据完整性

它**没有**检查 `require$$0`、裸 `require(`、`module.exports`、`exports.`、动态 `import(` 这类“构建成功但浏览器必炸”的残留。

# 问题判断

当前问题不是“某一行代码写错了”，而是 **userscript 构建目标与最终 bundle 包装形态不一致**：

- 目标要求：浏览器可直接执行的单文件 IIFE
- 实际结果：产物顶层 wrapper 和中部依赖代码都仍包含 CommonJS/Node 模块系统残留

因此，修复顺序必须非常严格：

1. 先确认 **Rollup 如何生成顶层 `require$$0/require$$1` wrapper**
2. 再决定是通过 userscript 专属打包配置修复，还是需要引入 `sql.js` 最小适配层
3. 禁止一上来就改共享业务代码或插件入口文件

如果直接粗暴合并入口、硬改插件逻辑、或者只在产物里做字符串替换，都会制造更大的隐患：

- 破坏插件模式初始化时序
- 污染 runtime mode 判定
- 引入后续不可维护的打包补丁

# 实施策略

采用**保留双入口架构，只定向修复 userscript 构建链**的方案。

核心原则：

1. **不合并 userscript / extension 入口。**
2. **不修改插件模式行为语义。**
3. **只对 userscript 分支处理 `sql.js`/CommonJS 残留问题。**
4. **补上构建后静态审计，防止坏 bundle 再次通过。**

# 实施步骤

## Step 1：先精确确认顶层 wrapper 的生成路径

在实施时先读取：

- `dist/index.bundle.js` 开头、`sql.js` 残留片段和文件结尾，确认顶层 wrapper 与依赖残留的对应关系
- `rollup.config.js` 中 userscript 分支的 CommonJS / nodeResolve 配置

目标：确认当前顶层 `require$$0/require$$1` wrapper 是哪种 Rollup 包装路径生成的：

- CommonJS helper 注入
- external 映射残留
- 依赖模块导出包装不完全

这个步骤不是形式主义，而是为了避免把“顶层 bundle wrapper 问题”误修成“入口文件问题”或“盲改 `sql.js` 业务接入问题”。

## Step 2：只针对 userscript 分支调整 Rollup 打包策略

优先检查并修正 `rollup.config.js` 中 userscript 分支的以下点：

1. `@rollup/plugin-commonjs` 的配置是否需要细化为 userscript 专用配置
2. 是否需要限制/指定 `include`，确保 `sql.js` 被完整转换而不是半 CommonJS 半外链
3. 是否需要开启或调整与混合模块相关的配置（例如 `transformMixedEsModules` 等）
4. 是否需要让 userscript 分支与 extension 分支使用不同的 CommonJS 插件实例，而不是共用同一套宽泛配置

目标是让 `dist/index.bundle.js` 最终变成纯浏览器可执行的单文件，不再残留：

- 顶层 `(function (require$$0, require$$1) { ... })(require$$0, require$$1);` 这类依赖外部绑定的 wrapper
- `require$$0`
- `require(`
- `module.exports`
- `exports.`

## Step 3：如果仅靠 Rollup 配置不够，再增加最小依赖适配层

若 Step 2 后顶层 wrapper 仍然由 `sql.js` 或其他依赖的 CommonJS 转换链触发，则在 `src/data/sqlite/sqlite-engine.ts` 附近增加**最小适配层**，原则如下：

1. 保持 `SqliteEngine` 对上层暴露的接口不变
2. 保持 `init()` 的错误处理和 fallback 语义不变
3. 避免改动插件启动链、runtime mode 判断链
4. 让 userscript 与 extension 可以继续共用业务接口，但允许在依赖解析层面有环境差异

适配层可能的方向：

- 将 `sql.js` 的导入封装到单独模块中
- 为 userscript 提供更明确的浏览器可打包入口
- 避免把 Node 侧残留（例如 `fs`）带进最终 IIFE

是否真的需要做到这一步，要以 Step 2 的结果为准，不能预设一定要改业务代码。换句话说，**先修 bundler，bundler 证明确实兜不住时，才动依赖接入层**。

## Step 4：补强构建产物审计

扩展 `scripts/audit-bundle.cjs`，新增 userscript 产物静态检查，至少覆盖：

1. `require$$0`
2. 裸 `require(`
3. `module.exports`
4. `exports.`
5. 裸动态 `import(`

这些检查必须针对 **userscript 单文件浏览器可执行性**，而不是继续停留在“语法合法就算过”的低标准上。

## Step 5：验证 userscript 与 extension 两条产线

实施后执行验证：

1. 构建 userscript，确认 `dist/index.bundle.js` 中不再出现 `require$$0` 等残留
2. 运行/审计 userscript 产物，确认浏览器环境可执行
3. 构建 extension，确认 `dist/extension/index.js` 仍可产出
4. 复核插件入口关键语义未变：
   - `_forceExtensionMode()` 仍在最前
   - `waitForTavernHelper()` 逻辑未变
   - `checkAndMarkInstance()` 逻辑未变
   - `mainInitialize_ACU()` 调用链未变

## 高优先级必查

- `rollup.config.js`
- `src/data/sqlite/sqlite-engine.ts`
- `dist/index.bundle.js`
- `scripts/audit-bundle.cjs`

## 高优先级只读验证（除非确有必要，否则不改）

# 风险与控制

## 风险 1：误伤插件模式

如果把 userscript 修复方案直接施加到共享入口或插件入口，会破坏插件模式时序。

控制方式：

- 修复优先落在 userscript 分支构建配置
- 插件入口文件仅用于回归校验，不轻易改动

## 风险 2：只消除表面报错，没有解决根因

如果只是移除 `require$$0` 文本，可能仍残留其他 CommonJS 包装碎片。

控制方式：

- 审计脚本新增一整组浏览器不可执行残留扫描
- 以“产物是否纯浏览器可执行”为验收标准，而不是“某个报错消失了”

## 风险 3：sql.js 修复引发 SQLite 初始化行为变化

如果依赖入口替换不当，可能改变 `initSqlJs()` 的返回语义或加载时机。

控制方式：

- 不改 `SqliteEngine` 对外接口
- 保留原有错误处理与 fallback 语义
- 在修复后重点验证 SQLite 初始化路径

# 验收标准

实施完成后，需要同时满足以下条件：

1. `dist/index.bundle.js` 是单 JS 文件，浏览器可直接执行
2. 产物顶层不再是依赖外部 `require$$0/require$$1` 绑定的 IIFE wrapper
3. 产物中不再出现 `require$$0` / 裸 `require(` / `module.exports` / `exports.`
4. userscript 运行时不再出现 `Uncaught ReferenceError: require$$0 is not defined`
5. extension 构建仍可成功输出
6. 插件模式以下语义不变：
- `_forceExtensionMode()` 的调用时机
- `waitForTavernHelper()` 的就绪与降级逻辑
- `checkAndMarkInstance()` 的互斥逻辑
- 共享 `mainInitialize_ACU()` 调用链

# 实施时的决策顺序

按这个顺序做，别乱跳：

1. 先确认 `dist/index.bundle.js` 中 CommonJS 包装的完整上下文
2. 再确认这个 wrapper 是由哪个 bundler 路径产生的
3. 然后改 userscript 构建配置
4. 配置不够再考虑 `sql.js` 最小适配层
5. 最后补审计并做双产线回归

这才叫方案。不是那种“先试试看改一处，也许就好了”的碰运气操作。那种做法，质量差得连侥幸都算不上。

# 新暴露问题：聊天身份识别初始化失败（构建修复后的下一层运行时问题）

## 新增关键对照事实：旧版 userscript 正常，当前合并版 userscript 异常，插件模式正常

助手补充的这条事实非常关键，必须单独写清楚：

1. 旧版单文件 `index.js` 在 userscript 模式下能正常读取聊天
2. 插件模式当前也能正常读取聊天
3. 只有**当前合并后的 userscript 单 JS 版**无法读取聊天

这直接说明：

- 不是宿主环境整体不给能力
- 不是插件模式本身坏了
- 而是**当前合并版 userscript** 相对旧版 userscript 丢了有效的聊天识别/宿主桥接行为

因此，后续实现不能只盯着当前 `init.ts` 自己的逻辑，还必须把“旧版 userscript 为什么能用”纳入对照组。

## 新问题现象

用户在打开聊天时出现如下日志：

```text
ACU: chatId still not available after 15000ms polling. Waiting for CHAT_CHANGED event.
```

同时插件/脚本界面显示“没有读到聊天文件”。

这不是和 `require$$0` 同类的构建错误，而是**构建修好之后，脚本终于跑到了更深一层的初始化逻辑，暴露出聊天身份识别入口不稳的问题**。

## 已确认事实

### 1. warning 的真实来源

已经精确定位到：

- `src/presentation/bootstrap/init.ts`

其中 `mainInitialize_ACU()` 在启动后执行以下逻辑：

1. 如果 `SillyTavern_API_ACU.chatId` 已可用，延迟 1 秒后调用：

```ts
await initWithChatId(SillyTavern_API_ACU!.chatId)
```

2. 如果 `chatId` 不可用，则每 200ms 轮询一次，最多 15 秒：

```ts
const chatId = SillyTavern_API_ACU?.chatId;
```

3. 如果 15 秒后仍不可用，只记录 warning：

```ts
ACU: chatId still not available after 15000ms polling. Waiting for CHAT_CHANGED event.
```

也就是说，当前所谓“主动获取当前聊天信息”的实现，实际上只是**反复读取同一个 `chatId` 字段**，没有真正使用第二来源主动探测当前聊天。这条注释和代码行为并不一致。

### 2. 当前聊天文件标识最终在哪里被设置

真正设置当前聊天文件标识的不是 `init.ts`，而是：

- `src/service/worldbook/injection-engine-state.ts`

其中：

```ts
export async function resetScriptStateForNewChat_ACU(chatFileName: string) {
  _set_currentChatFileIdentifier_ACU(cleanChatName_ACU(chatFileName));
  ...
}
```

这意味着系统真正依赖的是：

- 启动阶段拿到一个有效的 `chatId/chatFileName`
- 或收到一个有效的 `CHAT_CHANGED` 事件

只要这两件事都没发生，`currentChatFileIdentifier_ACU` 就不会被正确设置。

### 3. 当前默认状态会直接污染下游

- `src/service/runtime/state-manager.ts` 中：

```ts
export let currentChatFileIdentifier_ACU: any = 'unknown_chat_init';
```

如果初始化没有成功切换到真实聊天，下游多个模块都会继续拿这个默认占位值运行，包括：

- `service/worldbook/pipeline.ts`
- `service/settings/settings-service.ts`
- 多个 UI / plot / template 相关模块

所以“没有读到聊天文件”不是独立故障，而是**聊天身份识别失败后的自然后果**。

### 4. userscript / 插件模式的 API 读取方式本来就不同

`src/presentation/triggers/settings-ui-sync/settings-ui-connect.ts` 已经明确区分了两种模式：

- 插件模式：通过 `SillyTavern.getContext()` 的 Proxy 每次动态读取当前上下文
- 油猴脚本模式：直接使用 iframe 下扁平化的 `window.SillyTavern`

这说明“当前聊天识别”本来就不应假设两种模式都稳定依赖同一个字段表现。

### 5. 旧版 userscript 的宿主接入层比当前 TS 版更直接（仅作行为对照，不作回退模板）

旧版 `index.js` 已确认两段关键实现，但这里必须把边界写清楚：

- **它的价值是行为对照，不是实现来源**
- 当前项目整体已经完成重构，插件模式的宿主接入与初始化结构已经稳定
- 因此旧版代码只能帮助我们回答“当前 userscript 丢了什么能力”，不能直接回答“把哪段旧代码抄回来”

也就是说，后续实现如果直接把旧版 userscript 的宿主绑定、顶层桥接、初始化赋值整段搬回当前 TS 架构，就是把重构后已经建立的边界重新打碎。这不是修复，是倒退。

在这个前提下，再看旧版确认出的两段关键实现：

#### 旧版 `attemptToLoadCoreApis_ACU()`

位置：`index.js:17431-17450`

旧版实现直接通过：

```js
const parentWin = typeof window.parent !== 'undefined' ? window.parent : window;
SillyTavern_API_ACU = typeof SillyTavern !== 'undefined' ? SillyTavern : parentWin.SillyTavern;
TavernHelper_API_ACU = typeof TavernHelper !== 'undefined' ? TavernHelper : parentWin.TavernHelper;
```

也就是说，旧版 userscript 并没有引入当前 TS 版那种 userscript / 插件分流包装层，而是直接从 iframe 自身或顶层窗口取宿主对象。

#### 旧版 `mainInitialize_ACU()`

位置：`index.js:20773` 起，已确认初始化区包含：

- `ACU: Initializing with current chat on load: ${SillyTavern_API_ACU.chatId}`
- `await resetScriptStateForNewChat_ACU(SillyTavern_API_ACU.chatId)`
- 若失败则 `ACU: Could not get current chat ID on initial load. Waiting for CHAT_CHANGED event.`

同时旧版也在 `CHAT_CHANGED` 事件里调用：

```js
await resetScriptStateForNewChat_ACU(chatFileName)
```

这说明：

- 旧版和当前版的**最终状态设置点**其实相同，都是 `resetScriptStateForNewChat_ACU()`
- 真正差异更可能在**userscript 的宿主 API 接入方式**与**启动期获取当前 chatId 的时机/来源**

换句话说，如果后续实现只盯着 `resetScriptStateForNewChat_ACU()` 下游逻辑改，那基本就是修错层级。

### 6. 当前插件版 host API 逻辑才是后续实现的主参考

从当前 TS 代码已确认：

- `src/entry-extension.ts` 先 `_forceExtensionMode()`，再 `waitForTavernHelper()`，最后统一进入 `mainInitialize_ACU()`
- `src/presentation/triggers/settings-ui-sync/settings-ui-connect.ts` 中，插件模式并不是直接抓一个静态的 `window.SillyTavern`
- 插件模式实际通过 `SillyTavern.getContext()` + Proxy，在**每次属性读取**时获取最新上下文

这条链路非常关键，因为它说明新版重构后的主设计思路是：

1. **运行模式判断归 runtime-env 管**
2. **宿主能力接入归 host-api / settings-ui-connect 管**
3. **初始化编排归 init.ts 管**
4. **最终聊天身份落点统一归 `resetScriptStateForNewChat_ACU()` 管**

所以 userscript 修复的正确方向，不是回到旧版“直接从 parent/top 抓对象然后到处传”的做法，而是：

- 以插件版已经验证可工作的“动态宿主上下文读取”思想为主参考
- 在 userscript 分支补出**等价能力**
- 尽量把差异收敛在宿主接入层，而不是重新污染 bootstrap 和下游业务层

这才符合当前项目的重构方向。否则就是拿旧架构的成功，去破坏新架构已经整理出来的结构，短期看像修好了，长期只会制造新的耦合债。

## 问题判断

当前问题不是“聊天文件读取函数坏了”，而是**上游聊天身份识别入口设计过度依赖 `chatId` 和 `CHAT_CHANGED` 两个时序点**。

而在助手提供的对照事实下，这个判断还要再收紧一层：

- 问题很可能不仅是“依赖单一时序点”
- 还包括**当前合并版 userscript 的宿主 API 接入层，相比旧版 userscript 发生了行为变化**

所以真正需要对比的是：

1. 旧版 userscript 如何拿到 `SillyTavern_API_ACU`
2. 当前 TS userscript 如何拿到 `SillyTavern_API_ACU`
3. 当前插件版为什么能稳定读取运行时上下文
4. 两者在“启动时 chatId 是否可见”这件事上，到底差在哪

这四项里，**第 3 项优先级最高**。

原因不是因为插件模式可以直接复用，而是因为插件版代表的是当前重构后仍然成立的主干设计。userscript 修复必须优先向这条主干靠拢，而不是优先向旧版结构回退。

失败链路是这样的：

1. 启动时 `SillyTavern_API_ACU.chatId` 不可用
2. 15 秒轮询期间它仍然不可用
3. 初始 `CHAT_CHANGED` 事件可能已经错过，或当前场景根本不会再触发一次
4. `resetScriptStateForNewChat_ACU(validChatFileName)` 从未被调用
5. `currentChatFileIdentifier_ACU` 停留在 `unknown_chat_init`
6. 后续所有聊天文件读取都等价于“拿占位值去读真实聊天”

这类问题不能在下游打补丁。下游再怎么“容错”，也替代不了上游把当前聊天身份建立起来。

## 新增实施策略：修复聊天身份识别入口

### Step 6：补全聊天初始化链路侦察（以插件版主干逻辑为主参考）

实现前必须完整读取并确认：

- `src/shared/host-api.ts`
- `src/shared/runtime-env.ts`
- `src/entry-extension.ts`（插件主链路基线）
- `index.js`（旧版 userscript 可用实现，对照组）
- `src/presentation/bootstrap/init.ts`
- `src/service/worldbook/injection-engine-state.ts`
- `src/service/runtime/state-manager.ts`
- `src/presentation/triggers/settings-ui-sync/settings-ui-connect.ts`

目标是确认以下问题：

1. 当前插件版宿主接入层的稳定性具体来自什么：`getContext()`、Proxy、运行时模式锁定、还是其他约束共同作用
2. 当前 TS userscript 与插件版在宿主 API 接入方式上具体差了什么
3. 旧版 userscript 相比当前 TS userscript 多了什么有效能力，但这些能力能否在当前架构内以更小、更干净的方式重建
4. `SillyTavern_API_ACU.chatId` 在 userscript / 插件模式下分别来自哪里
5. 当前系统是否已经有第二来源可以主动拿到“当前聊天文件名/聊天身份”
6. `CHAT_CHANGED` 事件在当前场景下是没有触发，还是脚本注入时序晚于事件

### Step 7：先修 userscript 宿主接入层，再建立多来源聊天识别顺序

这一点必须写死，防止实现阶段偷懒：

- **优先修接入层，不优先修轮询层**
- 如果 userscript 当前拿到的 `SillyTavern_API_ACU` 本身就不稳定，那么在 `init.ts` 里给 `chatId` 增加再多轮询，基本都只是围着错误前提打转

因此，推荐顺序应改成：

1. 先核对/补强 `host-api / runtime-env / settings-ui-connect.ts` 这一层的 userscript 宿主接入方式
2. 让 `SillyTavern_API_ACU` 在 userscript 下具备尽可能接近插件版的“读取最新上下文”的能力
3. 然后再由 `init.ts` 消费这个更可靠的宿主上下文，建立启动期聊天识别
4. 最后才补充轮询结束后的降级与等待策略

修复时必须把“当前聊天识别”改成**多来源顺序探测**，而不是继续只轮询同一个字段。

但这里有一个额外约束：

- 多来源策略必须建立在“插件版主干逻辑 + 旧版 userscript 成功行为对照”这两个事实之上
- 插件版用于确定当前重构架构下应该保留的设计方向
- 旧版 userscript 用于识别当前 userscript 缺失的能力，不用于直接回退结构
- 不能脱离这两组事实，只在当前 TS 版里凭空叠更多轮询和判断

推荐顺序：

1. `SillyTavern_API_ACU.chatId`
2. 当前模式下可用的宿主上下文 API（例如插件版思路中的“动态上下文快照”在 userscript 下的等价来源）
3. 已有聊天元数据 / 当前聊天对象 / 当前聊天数组对应的宿主状态来源
4. `CHAT_CHANGED` 事件补偿

原则：

- 只要拿到一个有效、可清洗的聊天文件名，就立即统一调用：

```ts
await resetScriptStateForNewChat_ACU(validChatFileName)
```

- 禁止让下游模块自己猜当前聊天。

### Step 8：调整 15 秒轮询结束后的行为

当前逻辑在轮询失败后只是：

- 记一条 warning
- 被动等待未来某次 `CHAT_CHANGED`

这个行为标准太低，会把系统长期留在 `unknown_chat_init` 状态。

修复方向应是：

1. 轮询期间不仅检查 `chatId`，还检查备用来源
2. 轮询结束时，如果备用来源能提供当前聊天，就立刻初始化
3. 只有在**所有来源都确认不可用**时，才进入纯等待模式
4. 进入等待模式时，应明确把状态标记为“等待聊天身份建立”，而不是继续让下游像正常状态一样运行

### Step 9：保持插件模式不变量不变，并避免 userscript 修复反向污染插件结构

修复聊天身份识别时，以下逻辑仍然不能破坏：

1. `src/entry-extension.ts` 顶部 `_forceExtensionMode()` 的调用时机
2. 插件模式 `waitForTavernHelper()` 的就绪判断
3. `checkAndMarkInstance()` 的互斥逻辑
4. 两种模式最终都落到同一个 `mainInitialize_ACU()` 调用链

也就是说，**修复点应该集中在 `init.ts` 的聊天身份建立逻辑，不是重写整个启动架构**。

更准确地说，优先修复点应当是：

1. `settings-ui-connect.ts` / `host-api.ts` / `runtime-env.ts` 所代表的宿主接入层
2. 然后才是 `init.ts` 中对聊天身份的消费与降级逻辑

`init.ts` 不应成为第一个被打补丁的地方。否则就是上游接入没修好，先拿下游初始化逻辑兜锅，结构会越来越脏。

## 对实现阶段的额外约束

后续实现必须遵守以下顺序：

1. **先以当前插件版逻辑为主参考，确认重构后的宿主接入设计边界**
2. **再对照旧版 userscript 与当前 TS userscript 的宿主接入层，识别当前 userscript 缺失的有效能力**
3. 再决定当前 userscript 是应在现有 `host-api / settings-ui-connect / init.ts` 结构上增加等价能力，还是需要提炼旧版中某个最小桥接能力重建
4. **禁止直接回抄旧版 userscript 宿主接入代码**，除非能逐项证明该代码只是在当前架构中缺失的一项最小能力，并且已先论证无法用当前结构更干净地表达
5. 禁止因为“插件模式正常”就直接把插件模式的读法视为 userscript 的现成答案；插件版只能提供设计方向，不能替代 userscript 环境事实确认

原因很简单：

- 插件模式正常，说明当前重构后的主干设计本身成立
- 旧版 userscript 正常，说明 userscript 这条链路历史上具备过某种有效能力
- 当前修复的任务，是在**不破坏前者的前提下**，把后者缺失的能力重新补回当前架构

如果忽略这个对照组，后续实现很容易继续围着插件模式打转，最后把 userscript 问题修成另一个抽象层面的假象。

如果忽略“插件版主干逻辑优先”这个约束，后续实现也很容易走向另一种错误：

- 表面上 userscript 修好了
- 实际上是把旧版耦合结构重新埋回来了
- 最后插件模式和共享主链路都会被 userscript 的历史包袱反向污染

这种修法看似省事，实际维护成本高得离谱，不接受。

## 新增验证场景

实现后必须至少验证以下五类场景：

并且这五类验证必须额外附带一个总前提：

- **插件模式原有初始化与运行行为无回归**
- 包括 `_forceExtensionMode()`、`waitForTavernHelper()`、`checkAndMarkInstance()`、以及 `mainInitialize_ACU()` 主调用链保持原语义

1. **页面打开时已经处于某个聊天中**
   - 即使没有新的 `CHAT_CHANGED`，也能正确识别当前聊天

2. **脚本注入晚于初始 `CHAT_CHANGED` 事件**
   - 不能因为错过初始事件就永远停留在 `unknown_chat_init`

3. **正常切换聊天**
   - `CHAT_CHANGED` 仍然能正确驱动状态切换

4. **`chatId` 长时间缺失但备用来源可用**
   - 不能继续等满 15 秒然后什么都不做

5. **收到空值 / 非法聊天名**
   - 仍然保持现有保护逻辑，不允许把空值写进 `currentChatFileIdentifier_ACU`

## 额外风险提醒

这类问题最容易犯的错误，就是看到“没有读到聊天文件”，然后去改聊天文件读取函数本身。

那是典型的错误层级修复：

- 错在下游
- 原因在上游
- 后果是补丁越来越多、状态越来越乱、真正的初始化问题继续潜伏

正确做法是先把“当前聊天是谁”这个事实建立起来，再让所有下游自然恢复正常。
