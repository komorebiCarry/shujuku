我将按侦察→分析→执行→验收的流程处理这个任务。

## 侦察结论

- 模板预设切换入口在 `src/presentation/pages/popup-bindings-data.ts`：
  - 全局模板预设切换：第 503-520 行附近
  - 当前聊天模板预设切换：第 522-544 行附近
- 模板预设实际状态更新在 `src/service/template/template-preset-service.ts`：
  - `applyTemplatePresetToCurrent_ACU`
  - `applyTemplateSnapshotToScope_ACU`
- 当前聊天模板作用域切换在 `src/service/template/chat-scope/chat-scope-template.ts`：
  - `activateChatTemplatePresetSelection_ACU`
- 模板运行态应用在 `src/service/settings/settings-service.ts`：
  - `applyTemplateScopeForCurrentChat_ACU`
- 模板 UI 显示分散在多个 presentation 模块：
  - `src/presentation/components/template-preset-ui.ts`：模板预设下拉框与状态文案
  - `src/presentation/components/update-status-display.ts`：数据库状态卡片中的“当前生效模板预设”

- 侦察确认：模板切换成功后，业务状态和数据库运行态已经更新，但当前代码只局部刷新了模板 select，没有统一触发状态卡片与独立窗口刷新。

- 剧情推进预设切换入口在 `src/presentation/pages/popup-bindings-plot.ts`：
  - 全局剧情推进预设切换：第 333-347 行附近
  - 当前聊天剧情推进预设切换：第 349-370 行附近
- 剧情推进实际状态更新在 `src/service/plot/plot-logic.ts`：
  - `applyGlobalPlotPresetSelectionForEditor_ACU`
  - `switchCurrentChatPlotPreset_ACU`
- 剧情推进 UI 文案和下拉框刷新逻辑在 `src/presentation/pages/popup-helpers.ts`：
  - `loadPlotPresetSelect_ACU`
- 侦察确认：剧情推进切换成功后，当前代码只弹出 toast，不调用 `loadPlotPresetSelect_ACU`，也不触发数据库状态卡片或独立窗口刷新，因此已挂载 UI 会继续显示旧预设。

- API 路径也有同类问题：
  - `src/presentation/bootstrap/api-groups/template-preset-api.ts`
  - `src/presentation/bootstrap/api-groups/plot-preset-api.ts`
  它们在切换成功后没有统一刷新所有已打开 UI，导致脚本/API 触发切换时也可能留下旧显示。

## 根因判断

这次故障的根因不是数据库没切换，也不是预设没保存，而是**预设切换成功后缺少统一的 UI 同步出口**。

现状是：

1. service 层负责更新运行态、设置和聊天作用域；
2. presentation 层负责渲染下拉框、状态文案、数据库状态卡片和独立可视化窗口；
3. 但成功切换后，没有一个统一函数负责通知这些已挂载 UI 全部重渲染。

因此会出现：

- 数据库实际使用的新预设已经生效；
- 当前页面和独立窗口却仍显示旧预设；
- 只有关闭并重开界面，依靠初始化逻辑重新读取状态，显示才恢复正确。

## 实施方案

### 方案总览

新增一个 **presentation 层统一刷新入口**，用于在“模板预设 / 剧情推进预设切换成功后”同步所有当前已打开的 UI。然后让模板预设与剧情推进预设的：

- 手工 UI 切换入口
- API 切换入口

都复用这个统一入口。

### 具体执行步骤

#### step_1：新增统一的预设切换后 UI 刷新函数

在 presentation 层选择一个公共模块，新增类似“预设切换成功后刷新 UI”的函数。职责必须明确，至少覆盖以下四类刷新：

1. 模板预设 UI 重载
   - 调用 `refreshTemplatePresetSelectInUI_ACU` 或 `loadTemplatePresetSelect_ACU`
   - 刷新模板全局/当前聊天下拉框与状态文案

2. 剧情推进预设 UI 重载
   - 调用 `loadPlotPresetSelect_ACU`
   - 刷新剧情推进全局/当前聊天下拉框与状态文案

3. 数据库状态卡片刷新
   - 调用 `updateCardUpdateStatusDisplay_ACU`
   - 让“当前生效模板预设”等运行状态即时更新

4. 独立数据库编辑器刷新
   - 触发 `acu-visualizer-refresh-data`
   - 确保独立窗口顶部模板标识和编辑区数据同步

这个函数必须能安全处理“某些 UI 尚未挂载”的情况：弹窗不存在、窗口未打开时应静默跳过，不能抛错中断。

#### step_2：修复模板预设手工切换入口

修改 `src/presentation/pages/popup-bindings-data.ts`：

- 全局模板预设切换成功后，不再只调用局部 `refreshTemplatePresetUiState_ACU`
- 当前聊天模板预设切换成功后，也统一调用新的刷新入口
- 如果需要保留当前全局 select 选中值参数，保留；但最终刷新动作要统一收敛到公共入口

目标是让以下显示同时更新：

- 模板全局 select
- 当前聊天模板 select
- 模板状态说明文案
- 数据库状态卡片
- 独立数据库编辑器顶部模板标识

#### step_3：修复剧情推进预设手工切换入口

修改 `src/presentation/pages/popup-bindings-plot.ts`：

- `applyGlobalPlotPresetSelectionForEditor_ACU` 成功后调用统一刷新入口
- `switchCurrentChatPlotPreset_ACU` 成功后调用统一刷新入口

目标是让以下显示同时更新：

- 剧情推进全局 select
- 当前聊天剧情推进 select
- `plot-chat-scope-status`
- `plot-chat-origin-status`
- 数据库状态卡片（如果它依赖当前运行态）
- 任何已打开且依赖当前预设的编辑区显示

#### step_4：修复 API 切换入口

修改以下文件：

- `src/presentation/bootstrap/api-groups/template-preset-api.ts`
- `src/presentation/bootstrap/api-groups/plot-preset-api.ts`

在 API 切换成功后，同样调用统一刷新入口。

这样可以避免出现：

- UI 手工切换正常刷新
- 但脚本/API 切换后界面仍停留旧值

这种二次分叉。

#### step_5：必要时整理现有 UI 刷新函数的可复用性

如果统一入口在接入时发现：

- 模板侧只能在某个页面闭包里刷新，外部无法稳定调用
- 剧情推进侧的 `loadPlotPresetSelect_ACU` 或相关函数未导出/导出位置不合适

则做最小范围整理：

- 不改变现有业务语义
- 只整理导出方式和调用位置
- 让统一刷新入口可以稳定复用既有刷新函数

这里不能为了图快复制一份新逻辑。那种做法看着省事，实际是在复制下一次 bug。

#### step_6：补充测试

优先检查并扩展以下测试文件：

- `tests/service/template/template-preset-service.test.ts`
- `tests/service/plot/plot-state.test.ts`

如果现有测试基建适合测调用链，则增加对统一刷新入口的 spy/assert，至少验证：

1. 模板预设切换成功后会触发统一刷新入口
2. 剧情推进预设切换成功后会触发统一刷新入口
3. API 切换成功后也会触发统一刷新入口

如果 UI 层已有更合适的测试文件，则按现有测试组织方式落位，但目标不变：防止“切换成功但不刷新 UI”回归。

#### step_7：验证

执行与本问题直接相关的测试后，人工验证至少两个场景：

1. 当前聊天模板预设切换：
   - 不关闭数据库 UI
   - 数据库实际使用模板立即变化
   - UI 显示模板立即变化

2. 当前聊天剧情推进预设切换：
   - 不关闭数据库 UI
   - 运行态预设立即变化
   - UI 显示预设立即变化

如果独立数据库编辑器窗口可以同时打开，还要补验：

3. 编辑器保持打开时切换模板预设，顶部模板标识与内容即时刷新

## 边界条件清单

- 当前聊天模板预设切换为默认预设时，必须正确回退为“跟随当前全局/默认预设”的文案。
- 当前聊天模板预设在 `chat_override`、`preset_link`、`inherit_global` 三种模式之间切换时，状态文案必须同步变化。
- 当前聊天剧情推进预设切换为“跟随全局”时，当前聊天状态文案必须从“独立预设/历史快照”切回继承态。
- 当前弹窗未打开但独立数据库编辑器窗口已打开时，模板切换后仍必须刷新窗口。
- API 触发切换和手工切换必须表现一致。
- 某块 UI 不存在时不能抛错影响其余刷新。

## 风险说明

- 如果只在模板预设页面回调里补刷新，而不修剧情推进和 API 路径，这个问题会以另一条路径继续复现。
- 如果只刷新 select，不刷新状态卡片和独立窗口，用户仍会看到局部旧值，属于伪修复。
- 如果为模板和剧情推进分别复制两套刷新逻辑，后续继续演变会再次分叉，维护成本会越来越难看。

## 结论

这次修复应当围绕“统一 UI 同步出口”展开，而不是继续在单个页面里打零散补丁。数据库状态更新链路本身已经存在，缺的是一次明确、完整、可复用的 UI 刷新广播。
