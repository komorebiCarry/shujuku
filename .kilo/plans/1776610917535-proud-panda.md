# 优化剧情推进的任务级占位符、标签注入、历史回溯与任务级 API 预设

## 目标

基于当前项目真实实现，对剧情推进功能做四类改造，并确保 **油猴脚本产物** 与 **插件产物** 同步生效：

1. 将变量世界书条目触发依据从固定 `$6 + 固定上下文` 改为 **每个剧情任务自己的注入占位符与对应上下文**。
2. 在现有“提取标签”之外新增 **提取注入标签**，其优先级高于原提取标签，且 **即使未在最终注入中使用，也不能再自动追加到末尾**。
3. 修复剧情推进历史回溯仍可能读到 **已删除/已回退楼层的旧剧情推进数据** 的问题。
4. 为不同剧情任务新增 **任务级数据库 API 预设选择**；该配置 **保存到数据库自身设置/聊天设置链路，不保存到剧情推进预设模板**。

## 已确认事实

### 1. 双产物共用同一套源码，不是两份独立业务实现

- `rollup.config.js`
  - 油猴脚本：`src/index.ts -> dist/index.bundle.js`
  - 插件版：`src/entry-extension.ts -> dist/extension/index.js`
- `rollup.plus-assistantembedded.config.js`
  - 另一个插件构建：`src/entry-extension-plus-assistantembedded.ts -> dist/plus-assistantembedded/index.js`

结论：应优先修改 `src/**` 共享逻辑，再用构建验证双产物继承新行为。**不需要手工维护两套剧情推进实现。**

### 2. 剧情推进任务运行时的核心文件已定位

- `src/service/runtime/plot-runtime/plot-task-engine.ts`
  - `buildPlotSharedContext_ACU()` 构造 `$5/$6/$7/$8/$U/$C`
  - `$6` 当前固定来自 `getPlotFromHistory_ACU()`
  - 条件模板上下文 `plotContent` 当前也绑定到同一份 `lastPlotContent`
  - 单任务完成后调用 `extractPlotTagsFromResponse_ACU(rawResponse, normalizedTask.extractTags)`
- `src/service/runtime/plot-runtime/plot-tag-utils.ts`
  - 负责标签提取、占位替换、任务结果聚合、最终注入构建
  - 当前“提取标签”既参与聚合，也会在未被 `{{tag}}` 占位使用时自动追加到最终注入末尾
- `src/service/runtime/plot-runtime/plot-history-preset.ts`
  - 当前历史写入只保存 `qrf_plot` 与 `qrf_plot_preset`
  - 当前历史回溯只按 `qrf_plot / qrf_plot_preset` 检索，不具备任务粒度

### 3. 当前任务模型不具备此次需求所需字段

- `src/service/plot/plot-logic.ts` 中 `normalizePlotTask_ACU()` 当前只处理：

## 未确认项与处理原则

### 未确认项 1：历史幽灵数据的精确根因

已知现象是“删除楼层回到过去后，剧情推进仍可能读到已删除楼层的数据”。

目前已确认：

- 历史回溯在 `plot-history-preset.ts`
- 清理逻辑在 `chat-service.ts`
- 但尚未证明根因只来自 `getPlotFromHistory_ACU()` 本身，仍可能包含：
  - 聊天恢复后旧字段残留
  - 旧结构兼容回退过宽
  - 任务结果存储粒度过粗导致误读

**计划中的处理原则：**

1. 不只补一个 if 条件；
2. 要把“历史存储结构 + 回溯检索边界 + 清理策略 + 测试”一起收紧；
3. 若执行时发现还牵涉聊天恢复链路，再扩展修复范围，不用“后续再看”糊弄过去。

### 未确认项 2：任务级注入上下文的最终字段形态

用户需求已明确要“每个剧情推进任务自身使用的注入占位符号及上下文”。

已知需要：

- 不能再把变量世界书触发统一绑死在 `$6`
- 任务之间可能使用不同占位符
- 任务之间的上下文也可能不同

但具体字段命名与 UI 呈现方式，实施时需要根据当前任务模型与现有提示词编辑器风格决定。**计划先约束能力边界，不假装现在就知道最终字段名。**

### 已确认的产品约束：任务级 API 预设的持久化边界

根据用户回复：

- **任务级 API 预设保存到数据库自身设置/聊天设置链路**
- **不保存到剧情推进预设模板里**

这项已经是明确约束，不再作为待确认项。

## 方案设计

## 一、任务模型扩展与兼容迁移

### 目标

为每个剧情任务补齐独立配置，支撑占位符、历史回溯、标签注入和任务级 API 调度。

### 计划改动

修改：

- `src/shared/defaults-json.js`
- `src/service/plot/plot-logic.ts`

新增/补齐的任务级字段应至少覆盖：

- `extractInjectTags`：提取注入标签
- `taskApiPreset`：任务级 API 预设（默认空，表示继承全局 `plotApiPreset`）
- `plotInjectionPlaceholder`：任务自身的剧情注入占位符
- `plotInjectionContext` 或等价字段：任务自身的剧情注入上下文策略/来源标识

### 兼容要求

- 旧任务缺字段时自动补默认值
- 旧模板导入时不报错
- 旧聊天快照恢复时不丢失原任务结构
- 顶层旧字段（如 `plotSettings.extractTags`）继续维持兼容，但要通过主任务同步，而不是继续当作唯一真实来源

### 实现落点

- 在 `normalizePlotTask_ACU()` 中统一补齐新字段
- 在 `normalizePlotTasks_ACU()` 中确保旧数据数组加载后结构完整
- 在 `ensurePlotTasksCompat_ACU()` 中集中完成兼容迁移
- 在默认任务构造函数中给新字段明确默认值

## 二、任务级占位符与变量世界书触发改造

### 当前问题

当前剧情推进运行时把：

- 历史剧情内容固定绑定到 `$6`
- 条件模板的 `plotContent` 固定绑定到同一份 `lastPlotContent`

这会导致：

- 不同任务无法拥有自己的剧情注入来源
- 变量世界书条目触发仍被固定 `$6` 绑死
- 即使 UI 上将来新增任务级字段，没有运行时改造也只是装饰品

### 计划改动

修改：

- `src/service/runtime/plot-runtime/plot-task-engine.ts`
- `src/service/runtime/template-vars/seed-condition.ts`
- 可能涉及与世界书触发上下文拼装相关的运行时辅助模块（执行时补充）

### 目标行为

1. 每个任务可声明自己使用哪个剧情注入占位符。
2. 每个任务可声明自己的剧情注入上下文来源。
3. 变量世界书条目触发时，不再默认使用固定 `$6`，而是使用“当前任务解析后的剧情注入内容与上下文”。
4. 条件模板/关键词匹配的 `plotContent` 来源也必须与当前任务保持一致，不能出现“注入用的是任务 A，触发判断用的还是旧 `$6`”这种自相矛盾的状态。

### 推荐实现方向

在 `buildPlotSharedContext_ACU()` 之上新增“按任务解析注入上下文”的层，而不是继续把所有任务共享到同一个 `lastPlotContent`：

- 共享层保留全局基础上下文（表格、世界书、用户输入、AI 上下文）
- 任务层按 `plotInjectionPlaceholder + plotInjectionContext` 派生自己的剧情注入内容
- `renderPlotTaskContentWithIsolatedVariables_ACU()` / 条件模板求值使用任务级上下文对象，而不是只吃统一 `sharedContext.lastPlotContent`

### 历史结构要求

为了支撑任务级占位符，不应继续只存单一 `qrf_plot` 字符串。实施时应升级为：

- 任务/占位符维度的历史记录结构（名称可调整）
- 保留旧 `qrf_plot` 作为兼容回退读取来源

兼容顺序应为：

1. 先读新结构
2. 新结构缺失时，再按受控规则回退读旧 `qrf_plot`
3. 不允许旧字段无限兜底覆盖新逻辑

## 三、提取注入标签的新增与最终注入聚合重构

### 当前问题

`plot-tag-utils.ts` 当前的提取语义过于粗糙：

- `extractTags` 提取到的标签统一进入 `aggregatedTags`
- 若最终注入指令里存在 `{{tag}}`，已匹配的标签会被替换；未匹配的标签自动追加到末尾
- 若最终注入指令里没有 `{{tag}}`，全部标签都会追加到末尾

这与用户新需求冲突，因为用户要求新增一类标签：

- 逻辑上仍可被提取、传递、用于占位
- 但**即使没用到，也绝不能自动追加到最终注入末尾**

### 计划改动

修改：

- `src/service/runtime/plot-runtime/plot-tag-utils.ts`
- `src/service/runtime/plot-runtime/plot-task-engine.ts`
- `src/service/plot/plot-logic.ts`
- `src/shared/defaults-json.js`

### 新语义设计

每个任务的标签提取拆成两类：

1. `extractInjectTags`
   - 优先级更高
   - 同名标签若已在此类中提取，则普通 `extractTags` 中同名项失效
   - 可用于跨任务传递
   - 可用于最终注入中的 `{{tag}}` 占位替换
   - **若未被占位使用，不自动追加到最终注入末尾**

2. `extractTags`
   - 保持原有大部分逻辑
   - 可用于跨任务传递
   - 可用于最终注入中的 `{{tag}}` 占位替换
   - 若未被占位使用，仍按现有逻辑追加到最终注入末尾

### 数据结构建议

不要再让 `extractPlotTagsFromResponse_ACU()` 返回“提取结果 + injectedFragments”这种混合结构。应改为显式区分：

- `relayTags`：可传递标签集合
- `tailInjectableTags`：允许尾追加的标签集合
- `placeholderInjectableTags`：允许用于 `{{tag}}` 占位的标签集合

然后在最终聚合时：

- 先处理优先级覆盖（`extractInjectTags` 覆盖同名 `extractTags`）
- 再做 `{{tag}}` 占位替换
- 最后只把 **未使用且允许尾追加** 的标签拼到末尾

这样逻辑边界才清楚，不会继续把“能传递”和“能尾追加”混成一团。

## 四、历史回溯与删除楼层后的幽灵数据修复

### 当前问题

`getPlotFromHistory_ACU()` 当前只按：

- 当前聊天数组
- `beforeIndex / beforeUserInputHash / beforeUserInputText`
- `qrf_plot / qrf_plot_preset`

做回溯。

而 `savePlotToLatestMessage_ACU()` 只把剧情结果绑定到用户消息上。现有模型存在两个问题：

1. 存储粒度过粗，无法按任务/占位符区分；
2. 即便清理逻辑会删 `qrf_plot`，也不能保证所有回退/恢复场景下都不会读到旧值。

### 计划改动

修改：

- `src/service/runtime/plot-runtime/plot-history-preset.ts`
- `src/service/chat/chat-service.ts`
- `tests/service/runtime/plot-runtime/plot-history-preset.test.ts`
- `tests/service/chat/chat-service.test.ts`

### 目标行为

1. 剧情推进历史读写支持新结构（任务/占位符维度）。
2. 删除/清理旧楼层时，新旧结构都能被正确清理。
3. 回退到过去楼层后，历史回溯只读取当前有效聊天链路中的可用记录。
4. 旧 `qrf_plot` 仅作为兼容回退来源，且回退条件必须受限，不能继续把任何旧数据都当真。

### 推荐实现方向

- 升级历史写入结构：在用户消息上新增剧情任务结果映射字段（名称实施时确定）
- `savePlotToLatestMessage_ACU()` 同时维护：
  - 新结构
  - 旧 `qrf_plot/qrf_plot_preset`（仅在兼容窗口期保留）
- `getPlotFromHistory_ACU()` 改为支持按任务/占位符参数读取
- `purgeOldLayerData_ACU()` 清理时同步删除新历史字段
- 为测试补充以下断言：
  - `beforeIndex` 与任务级键共同工作
  - 删除旧楼层后新历史字段被清除
  - 新结构不存在时，才允许读旧结构

## 五、任务级 API 预设覆盖全局默认

### 已确认产品边界

这项配置：

- **保存到数据库自身设置/聊天设置链路**
- **不进入剧情推进预设模板导入导出**

### 当前问题

当前剧情推进只有全局 `settings_ACU.plotApiPreset`：

- UI 顶部有一个全局选择器
- 运行时 `callApi_ACU()` 固定读取它
- 任务级没有覆盖能力

### 计划改动

修改：

- `src/presentation/pages/main-popup-plot.ts`
- `src/presentation/components/plot-editors.ts`
- `src/presentation/pages/popup-bindings-plot.ts`
- `src/presentation/triggers/settings-ui-sync/settings-ui-api.ts`
- `src/service/plot/plot-logic.ts`
- `src/service/ai/api-call.ts`
- `src/service/runtime/plot-runtime/plot-task-engine.ts`

### 目标行为

1. 每个任务可以选择：
   - 继承全局剧情推进 API 预设
   - 使用指定 API 预设
2. 该配置保存在当前数据库设置/聊天设置链路中
3. 切换剧情推进预设模板时，**任务级 API 预设不跟随模板导入导出**
4. 删除某个 API 预设后，引用它的任务应安全回退为“继承全局”，不能残留非法引用

### 推荐实现方向

- 在任务模型中增加 `taskApiPreset`
- 在任务编辑区新增下拉框，选项：
  - `继承全局剧情推进 API 预设`
  - 所有现有 API 预设名
- 运行时在执行单任务前解析实际预设：
  - `task.taskApiPreset || settings_ACU.plotApiPreset`
- 调用路径改造优先级：
  - 任务级覆盖 > 全局剧情推进预设 > 当前 API 配置
- `refreshApiPresetSelectors_ACU()` 增强为同时支持刷新任务编辑器中的任务级 API 下拉

### 模板边界处理

实施时需要明确把 `taskApiPreset` 从“剧情推进预设模板快照”里剥离或排除，避免违反用户要求。

## 风险与验收标准

## 风险

1. **兼容风险**：旧模板、旧聊天快照、旧历史 `qrf_plot` 若迁移不完整，容易出现剧情推进失效或读错旧数据。
2. **语义串线风险**：如果标签传递与尾注入资格没有彻底拆开，`extractInjectTags` 很容易再次被旧聚合逻辑污染。
3. **持久化边界风险**：若任务级 API 预设被误纳入剧情预设模板，会直接违背用户约束。
4. **历史幽灵数据风险**：若只修读取不修写入和清理，删除楼层后仍可能留下隐形脏数据。

## 验收标准

实现完成后应满足：

1. 不同剧情任务可使用不同剧情注入占位符与上下文。
2. 变量世界书条目触发依据跟随当前任务，而不是固定 `$6`。
3. 新增“提取注入标签”后：
   - 同名标签优先覆盖旧提取标签
   - 可传递、可用于 `{{tag}}`
   - 未使用时不自动尾追加
4. 删除/回退楼层后，剧情推进不会再读取已删除楼层的数据。
5. 每个任务可单独选择数据库 API 预设；未选择时继承全局剧情推进 API 预设。
6. 任务级 API 预设不进入剧情推进预设模板导入导出。
7. 单测通过，且双产物构建通过。

## 实施备注

这次改动的真正难点不在 UI，而在 **历史结构、任务级上下文和标签聚合语义**。如果实施时试图只在表面加几个输入框，再拿旧 `$6` 和旧 `aggregatedTags` 继续硬撑，漏洞会明显得像是故意写给事故看的。

先立住数据模型，再收紧运行时，再接 UI，这才是像样的路线。
