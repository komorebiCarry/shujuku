# 优化插件版/js版手动填表前清空楼层数据

## 目标

在不破坏现有原生模式与 SQLite 模式共享架构的前提下，优化手动填表流程：

1. 手动填表前先弹出插件内确认框；
2. 用户确认后，先清除“本次手动填表实际要写入的 AI 楼层”上的全部表格数据；
3. 再执行新的手动填表；
4. 解决 SQLite 严格填表因旧楼层残留数据导致手动填表写不进去的问题；
5. 确保共享源码修改后，插件版与生成的 js 版产物都包含新逻辑。

---

## 已确认事实

### 1. 插件版和 js 版共用同一套源码

- `package.json` 中的 `build` / `build:extension` / `build:all` 都走 `rollup.config.js`
- `rollup.config.js` 已确认：
  - `src/index.ts` → `dist/index.bundle.js`
  - `src/entry-extension.ts` → `dist/extension/index.js`
- 因此本次不应把“插件版”和“js版”当成两套独立逻辑来计划；应修改共享 `src`，并在实现后验证两端产物都更新。

### 2. 手动填表入口和确认框基础设施已存在

- `src/presentation/pages/popup-bindings-status.ts`
  - 手动按钮绑定：`$manualUpdateCardButton_ACU.on('click', handleManualUpdate_ACU)`
  - 已引入 `showCustomConfirm_ACU`
- `src/presentation/theme/custom-confirm.ts`
  - 已有插件内自定义确认框实现，可直接复用

结论：不需要重新发明确认框组件，直接复用现有自定义确认框即可。

### 3. 手动填表真正的业务编排在 service 层

- `src/presentation/triggers/update-process.ts`
  - `handleManualUpdate_ACU()` 负责收集 UI 输入、调用 service、显示 toast
- `src/service/table/update-orchestrator.ts`
  - `orchestrateManualUpdate_ACU()` 负责：
    - 加载聊天消息
    - 刷新 merged data
    - 计算 `allAiMessageIndices`
    - 根据 `skipUpdateFloors` / `autoUpdateThreshold` 得到 `contextScopeIndices`
    - 按 `groupId` 把目标表拆分成多个 group
    - 调用 `processBatch(..., 'manual_independent', ...)`

结论：

- 确认框应放在 presentation 层；
- “清空楼层数据”动作应放在 service / repository 层；
- 不能把真实清理逻辑塞进 UI 文件里硬写字段删除。

### 4. “对应楼层”在现有实现里有明确技术含义

不是抽象上的“最近楼层”，而是：

- `processUpdatesBatch_ACU()` 中每个 batch 的 `finalSaveTargetIndex = lastMessageIndexOfBatch`
- `executeCardUpdateCore_ACU()` 成功后调用：
  - `saveIndependentTableToChatHistory_ACU(saveTargetIndex, ...)`

这说明：

- 每个 hand manual group / batch 实际写入的是某个明确的 AI 消息索引；
- 如果要“先清除对应楼层所有表格数据”，就必须按这些实际目标消息索引逐个清除；
- 只在手动更新开始前清一次“最后一层”会漏掉多 group 场景。

### 5. 原生模式和 SQLite 模式已经有统一抽象层

- `src/shared/table-storage-provider.ts` 定义 `ITableStorageProvider`
- `src/service/table/table-storage-strategy.ts` 负责按模式切 provider
- `src/service/table/native-table-service-adapter.ts` 封装原生模式
- `src/service/table/sql-table-service.ts` 封装 SQLite 模式
- `src/service/ai/prompt-builder/table-edit-parser.ts` 是两者共用入口：
  - SQLite 模式识别 SQL 内容后交给 provider.applyEdits()
  - 原生模式继续走 DSL 解析

结论：

- 本次优化不能只在 SQL 模式打补丁；
- 应优先把“手动填表前先清楼层”接在共享编排层，让两种模式统一受益。

### 6. SQL 严格填表失败的根因链已经明确

当前 SQLite 模式下：

- `SqlTableService.applyEdits()` 会基于当前 SQLite / JSON 视图执行严格 SQL
- 若目标楼层上已经存在旧表格数据，再次手动填表时，AI 生成的 SQL 可能因：
  - 旧行仍存在
  - 主键/row_id 冲突
  - 表状态与预期不一致
  - 旧数据残留导致 UPDATE/INSERT 条件不成立
  而失败
- 当前失败后只是把 SQL 错误反馈注入 prompt 重试，并不会先清空旧楼层数据

结论：用户提出的“先清空对应楼层，再重新填表”并不是附加功能，而是针对当前失败机制的直接修复路径。

### 7. 消息级表格数据存储结构已经确认

`src/data/repositories/chat-message-data-repo.ts` 已确认：

- 新版：`message.TavernDB_ACU_IsolatedData[isolationKey]`
  - `independentData`
  - `modifiedKeys`
  - `updateGroupKeys`
- 旧版兼容字段：
  - `TavernDB_ACU_IndependentData`
  - `TavernDB_ACU_Data`
  - `TavernDB_ACU_SummaryData`
  - `TavernDB_ACU_ModifiedKeys`
  - `TavernDB_ACU_UpdateGroupKeys`
  - `TavernDB_ACU_Identity`

同时已确认：

- 现有 `purgeSheetKeysFromMessage_ACU(msg, sheetKeys)` 只能“按表删”
- 用户需求是“清空该楼层的所有表格数据”

结论：

- 不能直接把 `purgeSheetKeysFromMessage_ACU` 当作完整解决方案；
- 需要先核对是否已有“整层清空”能力；没有的话就新增明确语义的 helper。

### 8. 清空后不能只改内存，必须走持久化与刷新链路

- `src/data/gateways/chat-gateway.ts` 提供 `saveChatToHost_ACU()`
- `src/service/table/table-storage-strategy.ts` 提供 `reloadStorageProvider()`
- `orchestrateManualUpdate_ACU()` 当前已有 `refreshData()` 回调，用于刷新 merged data

结论：

- 清空楼层消息字段后，必须保存聊天并刷新运行时数据；
- 否则 SQLite 可能继续拿旧内存数据库执行后续 SQL，清空动作就会变成表面文章。

## 风险与关键约束

### 风险 1：多 group / 多 batch 漏清楼层

如果只清一次“最后一层”，那么：

- 多个 group 对应不同 `finalSaveTargetIndex` 时会漏清；
- SQL 模式仍可能被未清的旧楼层数据卡住。

必须做法：

- 先根据所有 group 计算实际目标楼层索引；
- 去重后逐个清空。

### 风险 2：只清新版字段，旧版兼容字段残留导致旧数据复活

当前 merge 逻辑仍兼容旧版字段：

- `TavernDB_ACU_IndependentData`
- `TavernDB_ACU_Data`
- `TavernDB_ACU_SummaryData`

如果只删 `IsolatedData`，merge 时旧数据仍可能被重新读回。

必须做法：

- 清空动作必须覆盖新版 + 旧版兼容字段。

### 风险 3：把确认逻辑和清理逻辑混在 UI 层

后果：

- 页面绑定层会知道太多消息字段细节；
- 逻辑难测、难复用；
- 原生/SQLite 模式共享边界会被破坏。

必须做法：

- UI 只负责确认；
- service/repository 负责“按楼层清空表格数据”。

### 风险 4：清空后未刷新 provider / merged data

SQLite 模式是状态型 provider：

- 清空消息字段只是持久层变了；
- 若不 reload/refresh，运行时仍可能持有旧视图。

必须做法：

- 清空完成后立即走保存 + 刷新链路，再开始新的手动填表。

### 风险 5：原生模式默认提示词若仍使用 1 基对象列索引，会稳定导致手动填表错列

本轮补充核查已确认：

- `src/service/ai/prompt-builder/table-edit-parser.ts` 中，原生模式 `insertRow(tableIndex, dataObject)` 的对象写入逻辑使用 **0 基列索引**；
- 实现方式是遍历 `headers.forEach((_, colIndex) => ...)`，并从 `data[colIndex]` / `data[String(colIndex)]` 取值；
- 因此对象键：
  - `"0"` = 第一列数据
  - `"1"` = 第二列数据

这意味着：

- 如果默认提示词让模型生成 `{"1":"时间","2":"地点","3":"内容"...}`；
- 那么这些值会在原生模式下**整体右移一列**；
- 这不是偶发，而是当前解析规则下的必然后果。

同时已确认：

- `src/presentation/pages/popup-bindings-data.ts` 中已有一份正确说明，明确写了 `colIndex` 从 0 开始，示例为 `insertRow(10, {"0": "数据1", "1": 100})`。

结论：

- 当前项目内部很可能同时存在“正确的 0 基示例”和“错误的 1 基示例”；
- 真正要修的是**原生模式实际生效的默认提示词来源**，不是去改解析器做 1 基兼容。

必须做法：

- 找出所有实际参与原生模式手动填表 prompt 生成的示例文本；
- 将错误的对象列索引示例统一恢复为从 `"0"` 开始；
- 同步校对 UI 帮助文案与实际 prompt，避免出现一份写 0 基、一份写 1 基的自相矛盾状态。

## 实施方案

### Step 1：先核对是否已有可复用的“按楼层清空表格数据”能力

实现前先读：

- `src/presentation/triggers/data-admin-ui.ts`
- `src/service/chat/chat-service.ts`

目的：

- 确认是否已有“删除各楼层表格数据”的现成 service helper；
- 如果已有且语义符合“清空单个目标 AI 楼层全部表格数据”，直接复用；
- 如果只有批量清空、重置聊天、覆盖模板等相近但不等价能力，则不能凑合复用，需要新增专用 helper。

这一点很关键。助手如果不先核实，直接新造函数，很容易把已有逻辑绕开；反过来如果看见“差不多”的函数就硬复用，也一样会把语义搞错。

### Step 2：在 repository / service 层补齐“清空单个目标楼层全部表格数据”的通用能力

优先设计方向：

- repository 层：
  - 在 `src/data/repositories/chat-message-data-repo.ts` 新增明确语义 helper，负责清空一条消息上的全部表格存储字段；
  - 覆盖新版 `IsolatedData[isolationKey]` 和旧版兼容字段；
  - 不删除消息正文、不误删非表格业务字段。
- service 层：
  - 在更高一层封装“按 messageIndex 清空目标 AI 楼层表格数据并保存聊天”的 helper；
  - 集中处理 message 索引合法性、AI/user 消息判断、保存与刷新。

建议输出能力应满足：

1. 传入单个目标 messageIndex 即可清空；
2. 对无表格字段的消息视为成功空操作；
3. 可用于后续其他场景复用（如按楼层回滚、局部重算等）。

### Step 3：在 `orchestrateManualUpdate_ACU()` 中增加“预清空目标楼层”步骤

这是本次最关键的业务接入点。

做法：

1. 保留现有 group 计算逻辑；
2. 在进入 `for (const gKey of groupKeys)` 正式调用 `processBatch(...)` 前：
   - 根据每个 group 的 `indices` 计算其最终保存目标楼层；
   - 收集所有 `finalSaveTargetIndex`；
   - 去重；
   - 逐个清空；
3. 清空完成后：
   - `saveChatToHost_ACU()` 持久化；
   - 刷新 merged data；
   - 必要时 reload storage provider；
4. 然后再按现有流程执行新的手动填表。

注意：

- 不要把清空动作塞进 `executeCardUpdateCore_ACU()`；那里是通用执行核心，改进去会把非手动填表也污染到。
- 也不要每个 batch 执行前才临时清；那会让控制流更碎，失败恢复也更难整理。

### Step 4：在 `handleManualUpdate_ACU()` 中加入用户确认框

位置：`src/presentation/triggers/update-process.ts`

建议顺序：

1. 保留现有前置条件校验逻辑；
2. 在确认“确实能开始手动填表”之后，弹出 `showCustomConfirm_ACU()`；
3. 文案明确说明：
   - 将先清除本次对应楼层的全部表格数据；
   - 然后再执行新的手动填表；
   - 此举是为避免 SQL 严格填表逻辑因旧数据残留导致写入失败；
4. 用户取消：
   - 直接终止流程；
   - 不清空、不调用 AI、不刷新状态成“进行中”。

这里不要用模糊文案，例如“是否继续手动填表”。那种提示毫无技术信息，用户根本不知道系统要先删什么，差劲。

### Step 5：校准清空后的刷新链路，保证原生/SQLite 模式一致

实现时重点核对：

- 清空楼层后，仅 `refreshData()` 是否足够；
- SQLite 模式下是否还需要显式 `reloadStorageProvider()`；
- 原生模式是否只需重新 merge 即可。

推荐原则：

- 优先复用现有 provider reload + merged data refresh；
- 不新增分叉行为，避免 native/sql 各自一套补丁。

### Step 6：测试与验证

至少补以下验证：

1. **消息级清空逻辑**
   - 新版隔离字段存在时能清空
   - 旧版兼容字段存在时能清空
   - 同时存在时能全部清空
   - 空消息/无表格字段时为空操作

2. **手动更新编排逻辑**
   - 多 group 时会收集多个目标楼层并去重清空
   - 用户取消时不执行任何清空与填表
   - 清空后仍能继续走手动填表 happy path

3. **模式一致性验证**
   - 原生模式手动填表前也会先清楼层
   - SQLite 模式同样走这条共享路径

### Step 7：构建并验证两端产物

实现完成后：

- 构建共享源码；
- 验证 `dist/index.bundle.js` 包含新的确认文案 / 清空逻辑关键片段；
- 验证 `dist/extension/index.js` 同样包含相同逻辑；
- 防止再次出现“源码改了但插件版/js版没更新”的问题。

### Step 8：定位并修复原生模式手动填表提示词中的错误列索引示例

这是新增问题，优先级不低，因为它直接导致原生模式手动填表生成错列数据。

必须先做的核查：

1. 确认原生模式手动填表 prompt 的真实来源文件，而不是只看某个 UI 页面里展示出来的说明；
2. 确认项目内是否还存在旧示例使用 `{"1":...,"2":...}` 这种 1 基对象键写法；
3. 确认 SQLite 模式的 SQL 提示不受这次修复影响。

实现原则：

- 只修正原生模式提示词/示例文本，不修改 `table-edit-parser.ts` 的 0 基语义；
- 将原生模式对象示例统一为：
  - `{"0":"第一列数据","1":"第二列数据", ...}`
- 必要时补充一句明确说明：
  - `"0"` 表示第一列数据，不是从 `"1"` 开始。

这里不能偷懒去做“1 基兼容”。那种补丁只会把 DSL 语义搞脏，后面继续返工。

### Step 9：验证原生模式错列问题已消失

至少补以下验证：

1. 原生模式默认提示词中不再出现从 `"1"` 开始的对象列索引示例；
2. 原生模式按 `insertRow(..., {"0":"时间","1":"地点", ...})` 生成时不再整体右移；
3. SQLite 模式仍继续使用 SQL 提示，不受原生 DSL 文案修复影响；
4. 构建后的 `dist/index.bundle.js` 与 `dist/extension/index.js` 都包含修正后的提示词文本。

## 不应做的错误实现

以下做法都不可靠：

1. **只弹确认框，不做真实清空**
   - 这不是修复，只是加了个摆设。

2. **只在 SQLite 模式加特判**
   - 这样会破坏共享编排层，原生模式行为继续分裂。

3. **只清 `IsolatedData`，不清旧版字段**
   - merge 时旧数据仍可能复活。

4. **只清一个“最后楼层”**
   - 多 group 场景会漏清。

5. **在 UI 层直接 delete 一堆 TavernDB 字段**
   - 这会把数据操作逻辑打散到 presentation 层，后期只会更难维护。

6. **为了兼容旧错误示例，去修改原生 DSL 解析器支持 1 基对象键**
   - 这会让同一套 DSL 同时存在 0 基/1 基 两种语义，维护成本和歧义都会上升。

## 实现阶段完成标准

满足以下条件才算完成：

1. 手动填表前会弹出插件内确认框；
2. 用户确认后，系统会先清空本次实际目标 AI 楼层上的全部表格数据；
3. 原生模式与 SQLite 模式都遵循该流程；
4. SQLite 严格填表不再因为目标楼层旧数据残留而阻塞本次手动填表；
5. 取消确认时不会清空任何数据；
6. 共享源码构建后，`dist/index.bundle.js` 与 `dist/extension/index.js` 都包含新逻辑；
7. 原生模式默认提示词已统一恢复为从列 `"0"` 开始；
8. 原生模式手动填表不再因错误 1 基对象示例导致整体错列；
9. 至少补齐消息级清空、手动编排接入与原生模式错列修复的测试/验证。
