# 计划：填表表级 API 预设覆盖 + 剧情推进同组统一 API 预设

## 已确认事实

1. **插件版与打包 JS 版共享源码**
   - `src/entry-extension.ts` 明确说明插件入口与其他入口共享所有模块，只是启动方式不同。
   - `src/entry-extension-plus-assistantembedded.ts` 只是包装式追加 assistant，不改变主业务逻辑。
   - 结论：如果改动落在 `src/service` / `src/presentation` / `src/shared` 通用层，则插件版和打包 JS 版会同时生效。

2. **全局 API 预设机制已存在**
   - `settings_ACU.tableApiPreset`
   - `settings_ACU.plotApiPreset`
   - 预设列表由 `settings_ACU.apiPresets` 提供。
   - UI 刷新入口在 `src/presentation/triggers/settings-ui-sync/settings-ui-api.ts`
   - UI 绑定在 `src/presentation/pages/popup-bindings-data.ts`

3. **剧情推进已有任务级 API 预设覆盖，并且已明确不随模板导出**
   - `src/presentation/pages/main-popup-plot.ts` 第 159-166 行定义了 `plot-task-api-preset` 下拉框，文案写明“仅保存到数据库设置，不随模板导出”。
   - `src/presentation/components/plot-editors.ts` 中：
     - `loadCurrentPlotTaskToUI_ACU` 回填 `selectedTask.taskApiPreset`
     - `saveCurrentPlotTaskFromUI_ACU` 从 UI 读取并写回 `taskApiPreset`
   - `src/service/runtime/plot-runtime/plot-task-engine.ts` 已实现优先级：`taskApiPreset > plotApiPreset > 当前 API 配置`
   - 说明：用户要的“类似剧情推进任务里的该任务使用 API 预设配置功能”有现成模式可复用。

4. **填表当前没有表级 API 预设覆盖**
   - `src/service/template-assistant/service.ts` 直接使用 `settings_ACU.tableApiPreset`
   - `src/service/ai/prompt-builder/prompt-api-call.ts` 也是围绕整体 `tableApiPreset`
   - 说明：填表侧需要新增“表级覆盖”并贯穿 UI、settings、运行时。

5. **填表的“每个表格参数配置”UI 已锁定**
   - 用户所说的“每个表格的参数配置”对应 `src/presentation/pages/visualizer-main-config.ts` 的 `renderVisualizerConfigMode_ACU`
   - 其中“自动化更新参数”区块当前直接编辑 `sheet.updateConfig`
   - 已有字段包括：
     - `contextDepth`
     - `updateFrequency`
     - `batchSize`
     - `groupId`
     - `skipFloors`
     - `sendLatestRows`
   - 说明：如果把表级 API 预设也继续塞进 `sheet.updateConfig`，它会天然进入模板对象。

6. **填表存在“同组表一并更新”编排逻辑**
   - `src/service/table/update-orchestrator.ts`
   - 按 `templateData[sheetKey].updateConfig.groupId` 聚合 `updateGroups`
   - 同组表通过 `processBatch(... targetSheetKeys: group.sheetKeys ...)` 一并执行
   - 说明：真正应该做“统一使用对应 API 预设”的位置是 group 编排层，而不是只改单表调用点。

7. **模板清洗边界已经锁定到函数级别**
   - `src/service/template/chat-scope/chat-scope-template.ts`
     - `sanitizeTemplateSnapshotForChat_ACU` 会调用 `sanitizeChatSheetsObject_ACU`
   - `src/service/template/chat-scope/chat-scope-sheet.ts`
     - `sanitizeSheetForStorage_ACU` 只保留：
       - `name`
       - `uid`
       - `sourceData`
       - `content`
       - `updateConfig`
       - `exportConfig`
       - `TABLE_ORDER_FIELD_ACU`
     - `sanitizeChatSheetsObject_ACU` 对每个 `sheet_*` 调用该函数
   - 说明：任何新增到 `sheet.updateConfig` 的字段，默认都会进入模板保存、聊天快照和模板导出链路。

8. **模板相关入口已锁定**
   - `src/service/template/template-preset-service.ts`
     - `normalizeTemplateForPresetSave_ACU`：模板预设保存
     - `parseImportedTemplateData_ACU`：模板导入
     - `applyTemplateSnapshotToScope_ACU`：模板应用
     - `resolveTemplateForExport_ACU`：模板导出
   - 这些入口都依赖 `sanitizeTemplateSnapshotForChat_ACU` / `sanitizeChatSheetsObject_ACU`
   - 说明：靠“导出时特殊过滤某个 updateConfig 子字段”是危险假方案，会漏掉模板预设保存、聊天快照等其他链路。

## 已确认的关键结论

### 结论 1：填表表级 API 预设不能放进模板对象

这不是偏好问题，是代码事实决定的：

- `visualizer-main-config.ts` 当前直接编辑 `sheet.updateConfig`
- `sanitizeSheetForStorage_ACU` 会保留整个 `updateConfig`
- `sanitizeTemplateSnapshotForChat_ACU`、`normalizeTemplateForPresetSave_ACU`、`resolveTemplateForExport_ACU` 都会沿用这条链

所以只要把新字段塞进 `sheet.updateConfig`，它就一定会进入：

- 模板预设保存
- 聊天模板快照
- 模板导出
- 模板导入再应用

这与用户要求“同样不保存入模板中，只写进数据库插件设置里”正面冲突。

### 结论 2：填表表级 API 预设必须放进 settings 独立映射

用户已经明确要求：**不要按 `sheetKey`，改成根据表格名字来保存**。原因也已经给出：例如纪要表，不管模板怎么换，只要新的模板里还有“纪要表”，这个配置就应该继续保留。

因此推荐新增字段：

```ts
tableApiPresetOverridesByName: Record<string, string>
```

语义：

- key = **标准化后的表格名称**
- value = presetName
- 空字符串或不存在 = 使用 `settings_ACU.tableApiPreset`

这里的“标准化后的表格名称”至少应满足：

- `trim()` 去掉首尾空格
- 空字符串视为无效键
- 读写两端使用相同标准化规则

### 结论 3：表级 API 预设的 UI 入口应放在 visualizer 配置面板

即：

- `src/presentation/pages/visualizer-main-config.ts`

放在“自动化更新参数”区块最合理，用户认知最一致。

但它不能像现有字段一样写进 `sheet.updateConfig`，而必须通过当前表的**标准化名称**直接读写 `settings_ACU.tableApiPresetOverridesByName`。

### 结论 4：填表同组统一 preset 必须在 orchestrator 决议

不能在叶子调用点临时判断。正确位置是：

- `src/service/table/update-orchestrator.ts`

在 group 执行前，根据“本次触发更新的起始表”的**表格名称**读取其表级 preset，决议本次 group 的 `effective preset`，再传给整组下游调用。

### 结论 5：剧情推进不需要重做任务级 UI，但要补“同组统一”执行语义

UI 与保存链路已经存在：

- `main-popup-plot.ts`
- `plot-editors.ts`

真正需要补的是：

- `src/service/runtime/plot-runtime/plot-task-engine.ts`

把现有“逐任务单独回退”的逻辑提升为“同组统一 effective preset”。

## 目标拆解

### 目标 1：填表每表新增 API 预设配置
- 在每个表格的参数配置区域新增一个类似剧情推进任务里的 API 预设选择器。
- 空值表示“使用填表整体 API 预设配置”。

### 目标 2：该配置不保存进模板
- 新增字段不能写入模板对象、模板预设、聊天模板快照、模板导入导出结果。
- 只能写进数据库插件设置（settings）。

### 目标 3：按表格名称跨模板保留配置
- 只要新模板里仍存在同名表，就继续沿用该表的 API 预设覆盖。
- 不再按 `sheetKey` 绑定，因为 `sheetKey` 会随着模板切换而失去用户想要的稳定性。

### 目标 4：同组表统一使用该配置
- 如果从某表触发更新，而该表配置了表级 API 预设，则该表所在 update group 的所有同组表统一使用该 preset。
- 不应出现同一组里部分表走表级覆盖、部分表又退回整体配置的分裂行为。

### 目标 5：剧情推进补“同组任务统一使用”逻辑
- 现有任务级 `taskApiPreset` 逻辑需要扩展到同组任务执行语义。
- 同组任务应先决议组级 effective preset，再应用到组内任务执行。

## 推荐实现方案

### 方案核心

把“模板结构”和“运行时插件私有覆盖配置”彻底分层：

- 模板里仍然只保存表结构、字段、更新规则等模板本体
- 表级 API 预设覆盖改为保存到 `settings_ACU` 中的独立映射
- 这个映射的主键从 `sheetKey` 改为**标准化后的表格名称**
- 剧情推进继续使用任务级字段，但运行时增加“组级统一解析”
- 所有 effective preset 的最终决策都集中在编排层完成，不让每个叶子调用点各自猜

### 建议数据结构

在 settings 中新增：

```ts
tableApiPresetOverridesByName: Record<string, string>
```

优点：

- 不污染模板 schema
- 切换模板后，只要同名表仍存在，就能保留配置
- 更符合用户对“纪要表/人物表/总结表”这类业务名词的认知

代价：

- 同名表会冲突
- 改名会影响映射命中

这个代价必须明确写出来，不能装作不存在。

## 关键风险与约束

1. **不要把新字段塞进 `sheet.updateConfig`**
   - 这是最容易犯、也最恶心的错误。
   - 一旦混进去，模板预设、聊天模板快照、模板导出都会被污染。

2. **不要只在单表调用点做覆盖**
   - 用户明确要求同组表一起使用该配置。
   - 如果不在 orchestrator 的 group 决策点处理，这个需求就是假的完成。

3. **不要让剧情推进只保留单任务覆盖**
   - 用户明确要求“同组任务统一使用这个配置的逻辑加进去”。
   - 只保留现有 `taskApiPreset` 优先级而不做组级统一，属于没做完。

4. **删除 API 预设时必须清理新增引用**
   - 否则 UI 会出现已删除 preset 仍被引用的悬挂状态。

5. **按表名保存的代价必须明说**
   - 如果一个模板里有两个同名表，它们会共享同一个表级 API 预设覆盖。
   - 如果用户把表改名，原表名映射不会自动迁移，除非后续实现中额外补迁移逻辑。
   - 这不是 bug 修辞能抹掉的问题，而是“按表名作为业务主键”的天然代价。

## 当前结论

这次需求的正确落点不是“补两个下拉框”，而是：

- **填表：新增 settings 中按表名保存的表级覆盖映射 + visualizer 配置面板读写 + group 编排级 effective preset 决策**
- **剧情推进：复用现有任务级 UI/保存链路，把执行语义提升为同组统一**
- **模板边界：严格隔离，新增配置只进数据库插件设置，不进模板**

补充修正后，计划已经对齐助手的新要求：**不再按 `sheetKey`，而是按表格名字保存**。只要新模板里还有同名表，该配置就继续保留。这次总算不是拿技术主键强行压业务语义了。
