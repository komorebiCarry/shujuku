# 新增模式切换时提示恢复默认提示词功能（插件版 + JS版共用）

## 目标

在项目的插件版和最终 JS 版中新增同一个功能：

- 当用户在“原生模式（native）”和“SQL 模式（sqlite）”之间切换时；
- 弹出一个确认提示框，询问是否切换到该模式对应的默认提示词；
- 用户选择“是”时，自动恢复到目标模式对应的默认提示词；
- 用户选择“否”时，保持当前自定义提示词不变，只继续现有模式切换逻辑。

## 已确认事实

### 1. 模式切换的真实入口已经定位

- 文件：`src/presentation/pages/popup-bindings-status.ts`
- 位置：第 160-195 行
- 当前逻辑：
  1. 读取 radio 组 `name="${SCRIPT_ID_PREFIX_ACU}-storage-mode"`
  2. change 时拿到 `selectedMode`
  3. 直接把 `settings_ACU.storageMode = selectedMode`
  4. 调用 `saveSettingsAndNotify_ACU()`
  5. 调用 `switchStorageMode(selectedMode)` 执行 provider 切换和数据重载
  6. 成功则 toast，失败则回退设置和 radio

这说明：**新增确认弹框的最佳插入点就是这个 change 事件**。

### 2. 插件版和 JS 版共用同一条 popup 绑定链路

- `src/presentation/pages/popup-bindings.ts` 会统一调用 `bindStatusEvents_ACU()`
- 插件版入口和 JS 版入口最终都会走到这条 popup 事件绑定链路

因此这个功能不需要写两套：

> **只要改共用的 popup 模式切换事件，插件版和 JS 版都会一起获得这个功能。**

### 3. 存储模式字段和读取工具已明确

- 默认设置字段：`src/service/settings/settings-service.ts`
  - `storageMode: 'native' as const`
- 模式判断工具：`src/service/table/storage-mode.ts`
  - `getCurrentStorageMode()`
  - `isSqliteMode()`
  - `isNativeMode()`

### 4. 填表默认提示词已经存在双模式默认值

- 文件：`src/shared/defaults-json.js`
- 已确认常量：
  - `DEFAULT_CHAR_CARD_PROMPT_ACU` —— 原生模式默认填表提示词
  - `DEFAULT_CHAR_CARD_PROMPT_SQL_ACU` —— SQL 模式默认填表提示词

这意味着“切换到对应模式的默认提示词”并不是新概念，项目里已经有现成的数据来源。

### 5. 项目里已经存在“按当前模式恢复默认填表提示词”的函数

- 文件：`src/presentation/triggers/settings-ui-sync/settings-ui-config.ts`
- 函数：`resetDefaultCharCardPrompt_ACU()`
- 当前行为：
  - `settings_ACU.charCardPrompt = isSqliteMode() ? DEFAULT_CHAR_CARD_PROMPT_SQL_ACU : DEFAULT_CHAR_CARD_PROMPT_ACU`
  - 然后 `saveSettingsAndNotify_ACU()`
  - 然后 `loadSettingsAndRefreshUI_ACU()`

说明项目中已经承认“当前模式 -> 对应默认填表提示词”这条映射规则。

### 6. 项目里还存在同样按模式区分的“合并提示词”，但**本次需求不包含它**

- 文件：`src/shared/defaults-json.js`
- 已确认常量：
  - `DEFAULT_MERGE_SUMMARY_PROMPT_ACU`
  - `DEFAULT_MERGE_SUMMARY_PROMPT_SQL_ACU`
- 文件：`src/presentation/triggers/data-admin-ui.ts`
- 函数：`resetAllToDefaults_ACU()`
- 当前行为：
  - 会同时根据 `isSqliteMode()` 恢复 `settings_ACU.charCardPrompt`
  - 也会同时恢复 `settings_ACU.mergeSummaryPrompt`

但用户已经明确给出本次需求边界：

> 这里说的“对应的默认提示词”，指的是**点击“恢复默认提示词”按钮时恢复的那个提示词**。

结合当前代码，这个定义只对应：

- `settings_ACU.charCardPrompt`

不包含：

- `settings_ACU.mergeSummaryPrompt`

### 7. 当前项目现有交互方式支持直接使用原生 confirm

- 项目里已有多处直接使用 `confirm(...)`
- 也大量使用 `showToastr_ACU(...)`

所以本功能使用同步确认弹框并不违背当前代码约定，不需要为了一个二选一确认先造一套自定义模态框。

## 根因判断

当前问题不是“缺少模式切换能力”，而是：

> **模式切换只处理了存储 provider 和数据重载，没有给用户机会决定是否同步恢复到目标模式对应的默认提示词。**

结果就是：

- 用户切到 SQL 模式后，可能仍保留原生模式下的填表提示词；
- 用户切回原生模式后，也可能继续挂着 SQL 版提示词；
- 系统虽然技术上允许，但体验不一致，而且用户要手动去点“恢复默认”才能切回模式匹配的默认提示词。

## 实施原则

1. **功能必须落在插件版和 JS 版共用路径上**
   - 不分别改入口
   - 只改共用 popup 模式切换事件

2. **用户选择“否”时必须完全保持现有逻辑**
   - 不改提示词
   - 不额外刷新
   - 不影响模式切换成功/失败回退

3. **恢复默认动作不能抢跑**
   - 不能在 `switchStorageMode(selectedMode)` 之前就把提示词改掉
   - 否则模式切换失败时，提示词可能已经错切到目标模式默认值

4. **UI 必须同步刷新**
   - 否则内存里的提示词虽然变了，弹窗编辑区还显示旧内容
   - 用户会以为功能没生效

## 实施方案

### Step 1 — 在 `popup-bindings-status.ts` 的模式切换事件中加入确认流程

修改 `src/presentation/pages/popup-bindings-status.ts` 中 storage mode radio 的 `change` 事件：

- 在真正执行模式切换前，弹出确认框
- 文案建议包含：
  - 正在切换到“原生模式”或“SQL 模式”
  - 是否同时恢复到该模式对应的默认提示词
  - 选择“否”将保留当前自定义提示词
- 确认框只记录用户意图（要/不要恢复默认），**不立即写 settings**

### Step 2 — 将“按目标模式恢复默认填表提示词”抽成共用 helper

修改 `src/presentation/triggers/settings-ui-sync/settings-ui-config.ts`：

- 新增一个接收目标模式的 helper，例如：
  - `applyModeDefaultCharCardPrompt_ACU(mode: 'native' | 'sqlite')`
- 该 helper 内部只处理 `settings_ACU.charCardPrompt`，并根据 mode 执行明确映射：
  - native → `DEFAULT_CHAR_CARD_PROMPT_ACU`
  - sqlite → `DEFAULT_CHAR_CARD_PROMPT_SQL_ACU`

不要把这段映射逻辑硬编码复制到 `popup-bindings-status.ts` 里。那不是复用，是把未来维护成本埋进代码里。

### Step 3 — 模式切换成功后再执行默认提示词恢复

修改 `popup-bindings-status.ts` 中现有时序：

1. 用户选中目标模式
2. 弹出 confirm，记录是否恢复默认提示词
3. 继续现有：
   - 更新 `settings_ACU.storageMode`
   - `saveSettingsAndNotify_ACU()`
   - `await switchStorageMode(selectedMode)`
4. **只有在 switch 成功后**，如果用户前面选了“是”：
   - 调用 step_2 的 helper
   - 保存设置
   - 刷新 UI

这样才能避免“模式切换失败，但提示词已经被改到目标模式默认值”的状态错乱。

### Step 4 — 刷新提示词编辑区与提示文案

模式切换成功且用户选择恢复默认后，需要让 UI 立即反映结果：

- 复用已有 `saveSettingsAndNotify_ACU()`
- 复用已有 `loadSettingsAndRefreshUI_ACU()` 或更局部的提示词编辑区刷新函数
- 成功 toast 文案应明确说明：
  - 是否只切了模式
  - 还是同时恢复了对应默认提示词

例如：

- 只切模式：`已切换到 SQLite 模式！数据已重新加载。`
- 切模式 + 恢复默认：`已切换到 SQLite 模式，并恢复到该模式的默认提示词。`

### Step 5 — 补测试，覆盖用户选择与失败回退路径

实施阶段至少补/改以下测试：

1. 切换到 SQLite，用户选“是” → 填表提示词恢复为 `DEFAULT_CHAR_CARD_PROMPT_SQL_ACU`
2. 切换到原生，用户选“是” → 填表提示词恢复为 `DEFAULT_CHAR_CARD_PROMPT_ACU`
3. 用户选“否” → 当前提示词保持不变
4. `switchStorageMode(...)` 失败 → 提示词不会错误切换到目标模式默认值

## 已确认边界（不再作为未定项）

这里已经由用户明确拍板：

### “对应的默认提示词”只包含 `charCardPrompt`

也就是：

- 点击“是”后，只恢复与“恢复默认提示词”按钮当前行为一致的那套默认填表提示词
- 不恢复 `mergeSummaryPrompt`
- 不扩大到其他按模式分化的默认字段

## 验证方案

### 交互验证

1. 打开设置弹窗
2. 从原生模式切到 SQL 模式
3. 弹出确认框
4. 选“是” → 确认提示词编辑区显示 SQL 默认提示词
5. 再切回原生模式
6. 选“否” → 确认当前提示词保持不变

### 失败回退验证

强制制造 `switchStorageMode('sqlite')` 失败场景，确认：

- radio 回退
- `settings_ACU.storageMode` 回退
- 提示词不被错误改成 SQL 默认值

### 双端验证

- JS 版：验证 userscript / 最终单文件 JS 构建场景
- 插件版：验证 extension 场景

因为两端共用的是同一套 popup 绑定逻辑，所以重点是**确认共用实现两端都能触发**，而不是分别造两套实现。

## 实施完成标准

满足以下条件才算完成：

1. 用户切换原生/SQL 模式时会弹出确认框
2. 选“是”时，只恢复到目标模式对应的默认填表提示词（`charCardPrompt`）
3. 选“否”时，完全保持当前提示词与现有逻辑
4. 模式切换失败时，提示词不会错误切换
5. 插件版和 JS 版都通过共用路径获得该功能

## 备注

这次需求不难，但真正危险的点不是代码量，而是边界。尤其是“恢复默认”到底恢复哪些字段、在切换前还是切换后执行。如果这些不先说清楚，后面的实现就会在错误前提上自作聪明。

现在这个边界已经明确：

> 模式切换时点击“是”，只恢复“恢复默认提示词”按钮当前对应的那套填表默认提示词，也就是 `charCardPrompt`。

这个定义比我之前扩展到两组提示词更收敛，但它是用户明确确认过的事实，所以计划必须服从这个边界。
