# 修正剧情推进任务的变量世界书触发依据

## 一、问题重述（按当前确认后的真实需求）

助手之前指出得对：问题核心**不是**“给每个任务单独加一个专门的占位符配置项”。

真正要修的是这条错误逻辑：

- 当前剧情推进的变量世界书触发，**不管某个任务的 prompt 是否实际使用 `$6` 或任何剧情标签注入内容**，都会把“上一轮剧情推进内容”作为扫描文本的一部分参与本轮变量世界书触发。
- 这会导致世界书条目被**固定、过度、脱离任务实际输入**地触发。

助手要求的目标是：

- 每个剧情推进任务触发变量世界书时，**不再固定依赖上一轮整段剧情内容**。
- 触发依据应改为：
  1. **本任务 prompt 中实际使用的 `{{tag}}` 占位符所注入的标签内容**
  2. **本轮上下文**（近期聊天上下文 + 当前 userMessage）

这才是需求本体。继续围着“任务剧情注入占位符配置项”打转，会把实现带偏，属于前提错误。

## 二、已确认事实（代码证据）

### 1）当前世界书触发确实固定吃上一轮剧情内容

`src/service/runtime/plot-runtime/plot-task-engine.ts:113-116`

- 运行时先读取：`lastPlotContent = getPlotFromHistory_ACU(...)`
- 紧接着调用：`getWorldbookContentForPlot_ACU(plotSettings, userMessage, lastPlotContent)`

这说明剧情推进世界书扫描当前统一把 `lastPlotContent` 当作额外扫描文本传入。

### 2）世界书扫描文本当前由“近期上下文 + 当前 userMessage + 上轮剧情内容”构成

`src/service/runtime/plot-runtime/plot-task-engine.ts:571-622`

- `getWorldbookContentForPlot_ACU(..., extraBaseText)` 最终把：
  - `historyAndUserText`
  - `extraBaseText`
  拼成 `baseScanText`

其中当前调用方传入的 `extraBaseText` 就是 `lastPlotContent`。

### 3）变量世界书的触发完全依赖扫描文本命中关键词

`src/service/worldbook/pipeline.ts:808-850`

- `baseScanText` 会参与关键词匹配
- 递归触发时使用 `fullSearchText`
- 也就是说，只要把某段文本塞进 `baseScanText`，变量世界书就会基于它触发

所以现状已经坐实：

- **上一轮剧情推进内容当前被无条件纳入变量世界书触发依据**

### 4）当前任务 prompt 里 `{{tag}}` 的注入链路是独立存在的

`src/service/runtime/plot-runtime/plot-task-engine.ts:277-280`
`src/service/runtime/plot-runtime/plot-tag-utils.ts:203-259`

链路如下：

1. `getPlotPlaceholderTagNames_ACU(c)` 提取当前任务 prompt 中实际出现的 `{{tag}}`
2. `buildPlotTagMapFromText_ACU(sharedContext.lastPlotContent, requestedTagNames)` 从剧情历史文本里的 XML 标签中提取这些 tag 的内容
3. `replacePlotTagPlaceholders_ACU(c, relayTagMap)` 把 `{{tag}}` 替换成 `<tag>...</tag>`

这说明系统其实已经知道：

- **当前任务实际用了哪些 `{{tag}}`**
- **这些 `{{tag}}` 最终注入了什么内容**

但这套信息**没有被用于变量世界书触发**。这就是当前设计的断裂点。

## 三、问题本质

当前系统有两条彼此脱节的链路：

### A. prompt 注入链路（相对精确）

- 按任务实际写了哪些 `{{tag}}` 决定注入什么内容

### B. 世界书触发链路（过粗且固定）

- 不看任务是否实际使用这些标签
- 统一把整段上一轮剧情内容 `lastPlotContent/$6` 丢进世界书扫描文本

结果：

- 某任务就算完全没用 `$6`
- 也没用任何和上轮剧情有关的 `{{tag}}`
- 世界书仍然会被上轮剧情内容触发

这不是“设计风格不同”，这是触发依据和任务真实输入不一致，逻辑就是错的。

---

## 四、改造目标

将剧情推进任务的变量世界书触发依据改成：

1. **本轮上下文**
   - 近期聊天记录
   - 当前 userMessage

2. **当前任务 prompt 中实际使用到的 `{{tag}}` 注入内容**
   - 只针对当前任务 prompt 里真实出现的 placeholder
   - 只使用这些 placeholder 对应的标签内容

3. **不再默认把整段上一轮剧情内容作为变量世界书触发源**
   - 除非后续明确要求兼容兜底策略
   - 当前需求里不应再保留这个固定触发行为

## 六、实现方案（基于当前代码结构的可靠路径）

### 步骤 1：把世界书获取从“共享预计算”改成“任务级计算”

当前问题根源之一是：

- `worldbookContent` 在共享上下文初始化时就计算完成
- 当时根本不知道某个具体任务最终会使用哪些 `{{tag}}`

因此应改为：

- 在任务渲染阶段，按**当前任务**计算世界书内容
- 不再在共享上下文初始化时，用统一的 `lastPlotContent` 预先生成所有任务共用的世界书结果

建议调整点：

- `plot-task-engine.ts`
  - 缩减 `buildPlotTaskSharedContext`（或对应共享上下文初始化函数）中对 `worldbookContent` 的预取职责
  - 在 `renderPlotTaskMessages_ACU()` 或进入其之前，生成当前任务专属的世界书扫描文本并调用 `getWorldbookContentForPlot_ACU()`

### 步骤 2：新增“当前任务世界书扫描文本构建”函数

建议新增一个纯函数，职责明确，不要继续把逻辑揉在 `renderPlotTaskMessages_ACU()` 里，不然耦合会很脏。

建议能力：

- 输入：
  - 当前任务 promptGroup
  - `taskSharedContext.lastPlotContent / taskPlotContent`
  - 当前轮上下文（recentMessages + userMessage）
- 输出：
  - `placeholderTagNames`
  - `relayTagMap`
  - `worldbookTriggerText`

构造规则：

1. 扫描当前任务所有 prompt segment，提取全部 `{{tag}}`
2. 用这些 tagName 去当前任务剧情历史文本中取对应 XML 标签内容
3. 将这些实际命中的标签块拼成世界书触发文本的一部分
4. 再与“本轮上下文文本”拼接，形成最终 `worldbookTriggerText`

### 步骤 3：世界书扫描改为使用“任务实际注入标签 + 本轮上下文”

修改 `getWorldbookContentForPlot_ACU()` 的调用方式，而不是盲目重写世界书管线。

保留 `buildCombinedWorldbookContentByStrategy_ACU()` 不动，直接换它的 `baseScanText` 来源即可。

目标效果：

- 变量世界书扫描文本 = `historyAndUserText + taskInjectedTagBlocks`
- 不再默认拼接整段 `lastPlotContent`

其中：

- `historyAndUserText` 保留，用于满足“本轮上下文”要求
- `taskInjectedTagBlocks` 来自当前任务真实使用的 `{{tag}}`

### 步骤 4：任务 prompt 渲染与世界书触发共用同一份标签解析结果

这一步很关键，否则同一任务会出现：

- prompt 注入看一套 tag
- 世界书触发又重新算一套 tag

建议做法：

- 在任务渲染前先统一解析：
  - `placeholderTagNames`
  - `relayTagMap`
  - `injectedTagBlocks`
- prompt 渲染使用 `relayTagMap`
- 世界书扫描使用 `injectedTagBlocks`

这样才能保证：

- **世界书触发依据 exactly 等于任务实际使用的标签注入依据**

这才是助手要的行为一致性。

### 步骤 5：明确无 `{{tag}}` 的任务行为

这个边界不能含糊，否则实现时一定会有人偷偷把 `$6` 兜回来，结果又回到老路。

建议默认规则：

- 如果任务 prompt 中**没有任何 `{{tag}}`**：
  - 世界书触发仅使用“本轮上下文”（recentMessages + userMessage）
  - **不再额外拼接整段上一轮剧情内容**

这与当前需求完全一致，也最不容易再次引入“固定触发”。

## 九、实现边界与注意事项

### 已确认边界

- 当前 `{{tag}}` 的值来源，仍然是剧情历史文本中的 XML 标签内容
- 这次改造目标是让**世界书触发依据对齐任务实际注入内容**
- 不是重做整个标签系统

### 不应误做的事

1. **不要再新增一个“任务剧情注入占位符”输入项来解决这个问题**
   - 这不是需求重点
   - 就算加了，也不能自动修复世界书触发依据错误

2. **不要把“无 `{{tag}}`”任务偷偷回退成整段 `$6` 触发**
   - 这会把原 bug 变成“默认兼容行为”继续保留

3. **不要让 prompt 注入和世界书触发各算各的 tag 集合**
   - 否则结果还是会漂移

### 潜在风险

- 改为任务级获取世界书后，运行次数会增加
- 若世界书扫描成本高，可能影响剧情推进多任务模式的响应速度

可接受的处理方式：

- 先按正确性实现
- 若性能有问题，再考虑基于“任务 triggerText 哈希”的缓存

当前没有证据说明必须先上缓存，别还没修逻辑就先做性能杂耍，那是本末倒置。

## 十、最终实施结论

本次实现应围绕以下结论展开：

1. **删除剧情推进世界书对整段上一轮 `$6` 的固定触发依赖**
2. **删除之前误加的任务剧情注入占位符配置、UI、绑定、保存链路及相关模型字段**
3. **改为按任务实际使用的 `{{tag}}` 注入内容 + 本轮上下文触发变量世界书**
4. **让任务 prompt 渲染与世界书触发共用同一份标签解析结果**
5. **无 `{{tag}}` 时，只使用本轮上下文，不再追加整段上一轮剧情内容**

这版计划才真正落在助手刚刚澄清的需求上。上一版围着配置字段解释用途，结构看着完整，实际没有打中问题核心，不够合格。
