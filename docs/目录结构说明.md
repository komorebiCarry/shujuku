# 目录结构说明

## 目标

当前项目已经完成三轮模块化整理，当前阶段的核心原则仍然是：

- 保持 [`index.js`](../index.js) 作为稳定基线
- 将源码拆分到 [`src/`](../src/) 下按职责维护
- 通过 [`scripts/build-index.js`](../scripts/build-index.js) 按固定顺序拼接回单文件
- 先保证行为等价，再继续细分模块

当前状态可以概括为：

- 第 1 轮：单文件粗拆为三大板块
- 第 2 轮：建立 `core + ui + features + bootstrap` 分层
- 第 3 轮：将 [`src/features/`](../src/features/) 从历史上的“前中后切片”重组为“按业务域命名”

## 关于 UI、CSS 与功能层边界

这一轮最容易产生误解的点是：当前并不是把“所有与界面有关的东西”绝对纯化进 [`src/ui/`](../src/ui/)，而是优先完成功能域重组，并保持与 [`index.js`](../index.js) 全文等价。

可以这样理解：

- [`src/ui/`](../src/ui/) 放的是 **主界面主体**、**样式主体**、**窗口主体**、**visualizer 主体**
- [`src/features/`](../src/features/) 放的是 **主业务逻辑**
- [`src/features/ui/01_update_trigger.js`](../src/features/ui/01_update_trigger.js) 这类文件属于 **功能侧交互触发器**，不是主界面主体

关于之前提到的 CSS，需要特别说明：

- 真正的 visualizer 样式主体仍然在 [`src/ui/06_visualizer.js`](../src/ui/06_visualizer.js)
- 之所以在拆分分析时会提到 CSS，是因为原始 [`index.js`](../index.js) 的边界附近存在 visualizer 相关样式注释锚点
- 这属于“原始顺序与拼接边界识别”的问题，不代表这轮把 visualizer 主体又拆回 [`src/features/`](../src/features/)

因此，当前结构的准确理解应当是：

- **UI 主体已集中到 [`src/ui/`](../src/ui/)**
- **功能主链已重组到 [`src/features/`](../src/features/)**
- **少量历史边界痕迹仍被保留，以确保本轮拆分是安全拆分，而不是顺手重构**

- [`src/features/01_features_head.js`](../src/features/01_features_head.js)

这意味着当前结构已经进入新的稳定阶段：

- [`src/core/`](../src/core/) 保持稳定基础层
- [`src/ui/`](../src/ui/) 放主界面与样式主体
- [`src/features/`](../src/features/) 已经从“前中后切片”升级为“按业务域命名”
- [`scripts/build-index.js`](../scripts/build-index.js) 仍以原始执行顺序拼接
- [`dist/index.bundle.js`](../dist/index.bundle.js) 仍可作为模块化产物进行验证

### 下一步最推荐的模块化方向

当前最值得优先处理的，不再是 [`src/features/`](../src/features/)，而是 [`src/ui/05_main_popup.js`](../src/ui/05_main_popup.js)。

原因：
- 它仍然是当前最大的单体 UI 文件之一
- 同一个 [`openAutoCardPopup_ACU()`](../src/ui/05_main_popup.js:1) 中同时包含样式、导航、8 个分页内容、DOM 缓存、通用事件绑定、剧情推进绑定、正文替换绑定
- 它已经成为后续继续拆 UI 的主要阻塞点

### 推荐拆分策略

不建议第一次就把 [`src/ui/05_main_popup.js`](../src/ui/05_main_popup.js) 切成很多碎文件，而是先按“连续代码块 + 风险最低”的方式做一轮安全拆分。

推荐先重组为：

```text
src/ui/main_popup/
├─ 01_shell_and_update_tab.js
├─ 02_settings_tabs.js
├─ 03_import_tab.js
├─ 04_plot_and_optimization_tabs.js
├─ 05_common_bindings.js
├─ 06_plot_bindings_and_helpers.js
└─ 07_optimization_bindings_and_tail.js
```

对应含义：
- `01`：弹窗外壳、样式、导航、状态与更新页
- `02`：设置相关分页，如提示词、API、世界书、数据管理
- `03`：外部导入页
- `04`：剧情推进页与正文替换页的 HTML 主体
- `05`：通用 DOM 缓存与通用事件绑定
- `06`：剧情推进专属绑定与 helper
- `07`：正文替换专属绑定与弹窗尾部收束

### 为什么这样拆更稳

因为 [`src/ui/05_main_popup.js`](../src/ui/05_main_popup.js) 当前有几个高风险点：

- 大段 HTML 在同一个模板字符串里
- 很多逻辑共用 [`$popupInstance_ACU`](../src/ui/05_main_popup.js:2469)
- 剧情推进和正文替换不仅有独立分页，还有独立绑定与局部 helper
- 如果第一次就按“纯语义”切得太碎，容易破坏模板字符串边界、作用域边界和事件绑定顺序

所以更合适的顺序是：

1. 先做一轮安全拆分
2. 构建验证仍然保持与 [`index.js`](../index.js) 全文一致
3. 再继续细分成你想要的“设置界面 / 导入界面 / 更新界面”更纯粹的结构

在当前阶段，这套结构已经足以支撑第四轮围绕 [`src/ui/05_main_popup.js`](../src/ui/05_main_popup.js) 继续稳步模块化。
